# 10 · 决策记录与开放问题

本章记录 Loops 的**关键架构决策（ADR）**、**待决开放问题**与**风险登记**，供后续迭代与团队对齐。

---

## A · 架构决策记录（ADR）

### ADR-001 · 双 Agent 分工（Codex 大脑 / Claude Code 双手）
- **决策**：Codex 专司规划/拆解/审查/标注；Claude Code 专司实施。
- **理由**：① 利用不同厂商 agent 的相对优势；② 实施与审查角色分离，交叉校验，错误更早暴露；③ 规避"既写又审"的自洽盲区。
- **代价**：两套 agent 依赖、调用链更长。
- **状态**：采纳。

### ADR-002 · 文件优先的状态存储
- **决策**：`.loops/` 文件（JSON/YAML/MD）为真相源，SQLite 仅做派生索引。
- **理由**：可读、可 git、可重建、人可审计；契合"文档即真相"。
- **代价**：复杂查询需维护索引一致性。
- **状态**：采纳。

### ADR-003 · 每 Shard 一个独立 Claude Code 进程（上下文隔离）
- **决策**：每个 Shard 由全新、短命的 Claude Code 进程实施，上下文隔离。
- **理由**：**这是规避上下文压缩的核心手段**——进程结束即销毁上下文，从结构上消除"长上下文"。
- **边界**：该方案能缓解单个 Claude Code 会话过长导致的压缩/遗忘，但不能单独保证跨 Shard 一致性、集成正确性、测试覆盖与最终质量；必须配合接口契约、测试矩阵、测试证据与整体复查。
- **代价**：进程启动开销、需把必要上下文注入每个 Shard（自包含要求），还需维护全局状态与测试闭环。
- **状态**：采纳。

### ADR-004 · Codex 承担标注
- **决策**：所有标注由 Codex 生成与刷新（含终态标注）。
- **理由**：标注依赖"全局视图 + 审查结论"，与 Codex 大脑角色一致；统一来源避免多写者冲突。
- **代价**：Codex 工作量增加；需保证标注维度完整。
- **状态**：采纳。

### ADR-004B · Codex 承担测试负责人职责
- **决策**：Codex 不仅作为产品/架构师进行规划、拆解、审查和标注，也作为测试负责人生成 Test Matrix、审查 Test Record、判断测试缺失/失败/flaky，并把测试结论纳入收敛判据。
- **理由**：单 Shard 实现审查容易漏掉跨模块回归和测试缺口；如果没有测试证据，Codex 审查只能依赖 Claude Code 自述或 diff 摘要，可信度不足。测试矩阵让验收标准可执行，测试证据让审查可审计。
- **落地**：新增 `CodexAdapter.designTests()`、`CodexAdapter.reviewTests()`、`TestModule`、`Test Matrix`、`Test Record`、`test_status` 标注；必须测试缺失或失败时 Shard 不得 `DONE`。
- **状态**：采纳。

### ADR-005 · 唯一强制人工门禁 = Spec 审核
- **决策**：仅"产品文档审核"为默认强制门禁，其余环节自治。
- **理由**：在"自治效率"与"方向可控"间取平衡——方向错了越早拦截成本越低，而 Spec 是方向的载体。
- **代价**：实施阶段无人逐项把关（由 Codex 审查补偿）。
- **状态**：采纳（可在 config 增加额外门禁开关）。

### ADR-006 · 编排与 agent 解耦（Adapter 模式）
- **决策**：编排层只面对 Adapter 接口，agent 可替换。
- **理由**：抗 provider 变动；便于实验不同模型组合。
- **状态**：采纳。

### ADR-007 · 产品形态：Web 控制台 + 后端编排服务优先
- **决策**：Loops 采用 **Next.js Web 控制台 + NestJS API/Orchestrator + Runner Worker** 的产品形态；CLI 保留为本地执行和运维入口。
- **理由**：用户明确希望使用 NestJS + Next.js 实现前后端；SSO 登录、Web Issue 入口、Spec 审核、状态看板、标注看板、人工介入都天然需要服务端与 Web 控制台支撑。
- **代价**：MVP 工程复杂度高于纯 CLI；需要处理 Webhook、用户映射、权限、数据库索引。
- **状态**：采纳（已更新）。

### ADR-008 · Git 集成：commit-per-shard + 收敛 PR
- **决策**：每个 Shard 实施完成即产生**一次提交**（commit message 关联 `shard-id`）；收敛后聚合为本 Loop 的**一个 PR**，PR 描述附**标注摘要**，供人最终合并。
- **理由**：commit-per-shard 让每步可回溯、可单独 revert、可定位到 Shard；收敛 PR 让人做最终把关，契合"人在环"；标注摘要让 PR 审阅有据可循。
- **代价**：提交粒度变细、需维护分支策略。
- **状态**：采纳（已确认）。

### ADR-009 · Codex provider：OpenAI Codex CLI
- **决策**：大脑角色（规划/拆解/审查/标注）采用 **OpenAI Codex CLI**（`codex` 命令）。
- **理由**：用户确认；其作为可程序化调用的编码/推理 agent，适配"大脑"职责。
- **状态**：采纳（已确认）。经 Adapter 抽象，后续可替换为其他规划型 agent。

### ADR-010 · 开发期 Issue 入口：Web 页面优先，飞书后置
- **决策**：开发期 Loops 的一等 Issue 入口是 Web 控制台表单；GitHub Issue 仅作为可选同步目标，飞书入口作为后续集成。
- **理由**：用户明确指出前期开发不需要复杂飞书 Issue，先在页面提交即可；这能降低飞书开放平台、Webhook、消息卡片、用户映射的前置复杂度，更快验证核心闭环。
- **落地**：新增 `WebIssueIntake`、`source_channel=web`、`submitter`、`raw_payload_ref`；信息不足时在页面字段级提示补齐。
- **代价**：早期不能直接从飞书聊天创建 Issue；后续需补 `FeishuAdapter`。
- **状态**：采纳（覆盖上一版飞书优先决策）。

### ADR-011 · 工程脚手架与落地目录
- **决策**：后续创建项目使用 `npx create-dofe-ai vibecoding.dofe.ai`，技术栈为 NestJS + Next.js，项目目录固定为 `/Users/techwu/Documents/codes/dofe.ai/vibecoding.dofe.ai`。
- **依据**：已确认 `create-dofe-ai` npm 包当前 latest 为 `0.1.27`，描述为从 DofeAI monorepo scaffold 创建项目，关键词包含 `nextjs`、`nestjs`、`monorepo`。
- **状态**：采纳。

### ADR-013 · 统一登录源：Dofe SSO
- **决策**：Loops 不自建账号密码体系，唯一登录源为 `sso.dofe.ai`，用户信息与权限校验通过 `api.sso.dofe.ai`。
- **理由**：复用 Dofe 统一身份体系，减少账号、安全、权限维护成本；所有 Issue、审核、人工介入都能绑定到明确用户。
- **落地**：新增 `AuthModule` 与 `SsoAdapter`；所有写操作必须有 SSO 用户身份；开发期可提供 mock user 模式但不可作为生产登录源。
- **状态**：采纳。

### ADR-012 · 开源项目借鉴策略
- **决策**：不直接依赖单一开源项目，而是吸收多个开源项目的架构模式并产品化重组。
- **借鉴映射**：
  - **OpenAI Symphony**：借鉴 issue tracker 驱动、隔离 workspace、长运行 orchestration service 的思想。
  - **GitHub Spec Kit**：借鉴 Spec → Plan → Tasks → Converge 的文档化拆解流程。
  - **AIRunX / Agent Orchestrator / Baton**：借鉴 Git worktree、Issue 轮询、agent runner、PR/审查循环。
  - **Claude Squad / AWS CLI Agent Orchestrator**：借鉴多 CLI agent 会话隔离、tmux/worker 管理、人可介入。
  - **Codex Action / Claude Code Action**：借鉴 PR 审查、结构化输出、自动评论与状态回写。
- **理由**：现有开源项目没有完整覆盖"Web/飞书入口 + Codex 文档大脑 + 多 Claude Code 实施 + Codex 审查标注 + 循环回环"；组合借鉴比强绑定单项目更稳。
- **状态**：采纳。

## C · 风险登记

| 风险 | 等级 | 缓解 | 责任阶段 |
|------|------|------|---------|
| 上下文压缩未充分规避 | 高 | 预算硬约束 + 隔离 + 自包含校验 | M1 验证 |
| 拆解质量不稳定 | 高 | 拆解自检 + 人工审核门禁 + 可重拆 | M1/M2 |
| Codex 审查能力不足（漏判） | 中 | 测试矩阵 + 测试证据 + 双 verdict 容忍 NEEDS-WORK 重做 + 整体复查兜底 | M1/M3 |
| 测试覆盖不足导致假收敛 | 高 | 必须测试项阻断 DONE；全局回归；终态标注检查 `test_status` | M1/M3 |
| 成本失控 | 中 | 熔断 + 上限 + 告警 | M4 |
| provider 不可用/限流 | 中 | Adapter 可替换 + 重试退避 | 全程 |
| SSO 对接不稳定 | 中 | M0 提供 mock user 开发模式；M1 接入真实 `sso.dofe.ai` | M0/M1 |
| 飞书 Webhook 不可达或验签失败 | 低 | 飞书后置，不阻塞 MVP；接入时使用本地隧道或可信公网服务 | 后续 |
| 并发改动冲突 | 中 | DAG + 文件白名单 + 串行降级 | M2 |
| 无限回环 | 低 | 最大回环次数 + 人工介入 | M3 |
| 安全（agent 越权/密钥泄露） | 中 | 沙箱 + 白名单 + 密钥隔离 | M2 起 |
