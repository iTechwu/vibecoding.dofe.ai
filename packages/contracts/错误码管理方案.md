# 错误码管理优化方案

## 📋 概述

本文档描述了将错误码系统从数字枚举（numeric enum）迁移到字符串枚举（string enum）的完整方案，以确保在打包（pack）过程中不会出现类型错误，并统一错误码管理架构。

## 🎯 优化目标

1. **统一错误码格式**：将所有数字枚举改为字符串枚举，避免打包时的类型问题
2. **完善错误码管理**：确保所有错误码都通过 `@packages/contracts/src/errors` 统一管理
3. **清理过期文件**：删除已迁移的旧文件，保持代码库整洁
4. **保持向后兼容**：在迁移过程中保持 API 响应的兼容性

### ⚠️ 待优化的问题

1. **数字枚举问题**
   - ❌ 所有域错误码使用 `enum` 定义，值为数字（如 `TeamErrorCode.TeamNotFound = 100404`）
   - ❌ 在打包过程中可能出现类型推断问题
   - ❌ 数字枚举在序列化/反序列化时可能丢失类型信息

2. **旧文件残留**

   - ❌ `error-response.ts` 中 `ErrorDetailSchema` 的 `errorCode` 定义为 `z.number()`
   - ❌ 需要改为字符串类型以匹配新的枚举格式

### 阶段一：错误码枚举字符串化（核心优化）

#### 1.1 更新域错误码定义

**目标文件**：`packages/contracts/src/errors/domains/*.errors.ts`

**变更内容**：

- 将所有 `enum` 改为字符串枚举（const object with string values）
- 保持错误码的数值语义（作为字符串值）
- 更新相关的类型映射

**示例变更**：

```typescript
// ❌ 旧格式（数字枚举）
export enum TeamErrorCode {
  TeamNotFound = 100404,
  TeamOpNoPermission = 101403,
}

// ✅ 新格式（字符串常量对象）
export const TeamErrorCode = {
  TeamNotFound: '100404',
  TeamOpNoPermission: '101403',
  // ... 其他错误码
} as const;

export type TeamErrorCode = (typeof TeamErrorCode)[keyof typeof TeamErrorCode];
```

#### 2.2 更新 API 响应格式

**决策点**：API 响应中的 `code` 字段应该保持为数字还是改为字符串？

**建议**：

- **保持向后兼容**：API 响应的 `code` 字段保持为数字（用于 HTTP 状态码兼容）
- **错误详情使用字符串**：`error.errorCode` 使用字符串格式（用于类型安全）

**示例响应格式**：

```json
{
  "code": 200, // HTTP 状态码（保持数字）
  "msg": "Team not found",
  "data": null,
  "error": {
    "errorCode": "100404", // 错误码（字符串）
    "errorType": "teamNotFound",
    "errorData": { "teamId": "123" }
  }
}
```
