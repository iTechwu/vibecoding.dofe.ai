# 07 · 集成与接口

本章定义 Loops 如何**程序化地调用 Codex 与 Claude Code**：调用契约、Prompt 模板、上下文隔离、容错与安全。核心原则——**编排层只面对 Adapter 接口，不感知具体 agent 的命令行细节**。

> 说明：本章给出的是**接口契约与建议命令形态**。具体 CLI 的 flag（如 `codex`、`claude` 的非交互参数）以实现时所用版本为准，Adapter 内部封装，不外泄到编排层。

---

## 1. Adapter 统一接口

```
CodexAdapter:
  plan(issue, repo_ctx)            → Spec
  decompose(spec, budget)          → { shards[], dag }
  designTests(spec, shards)        → TestMatrix
  reviewTests(test_record, matrix) → { test_verdict, issues[], fix_instructions[] }
  review(shard, impl_record)       → { verdict, issues[], fix_instructions[] }
  reviewGlobal(spec, all_records)  → { verdict, issues[] }
  annotate(targets[])              → annotations[]

ClaudeCodeAdapter:
  run(shard, project_root, sandbox) → impl_record

SsoAdapter:
  verifySession(request)            → authenticated_user
  getUserProfile(user_id)           → user_profile
  getUserRoles(user_id)             → roles[]

WebIssueIntake:
  submitIssue(form, user)           → issue_intake
  validateIssue(form, user)         → validation_result

FeishuAdapter:
  verifyEvent(request)              → verified_event
  parseIssueMessage(event)          → issue_intake
  sendClarification(issue, question)→ message_result
  sendStatus(issue, status_card)    → message_result
  resolveUser(feishu_user)          → loops_user

GitAdapter:
  createWorktree(loop, shard?)      → worktree
  commitShard(shard, files)         → commit
  createConvergencePR(loop)         → pr
```

每个方法内部 = 构造 Prompt → 调子进程 → 解析输出 → 校验 schema → 返回结构化结果。

## 2. CodexAdapter · 六类调用

Codex 以**非交互（headless）模式**被调用，输入为构造好的 Prompt + 必要上下文，输出要求为结构化（JSON/YAML）便于解析。

### 2.1 调用契约（通用）
- 输入：System Prompt（角色与约束）+ User Prompt（任务 + 上下文）+ 输出 schema 要求。
- 输出：严格符合 schema 的结构化结果（失败则重试/报错）。
- 超时：每类调用独立配置（生成较长，审查中等，标注较短）。
- 可观测：每次调用记入 `log.jsonl`（输入摘要、输出摘要、耗时、token）。

### 2.2 六类 Prompt 模板（要点）

**① plan（生成 Spec）**
- System：你是资深架构师，输出完整、可实施、可审核的产品文档。
- User：Issue 原文 + 代码库现状摘要 + Spec schema。
- 约束：必须含范围（含不做项）、方案、**可勾选验收清单**、风险、拆解指引。
- 输出：Spec（符合 [05 §2.2]）。

**② decompose（拆解）**
- System：你是任务拆解专家，目标是产出自包含、不超上下文预算的实施单元。
- User：Spec + `context_budget` + Shard schema + 拆解维度清单。
- 约束：每个 Shard `est_context < budget`；标注 `depends_on`；合并须覆盖 Spec 全部验收项。
- 输出：`{ shards[], dag, decomposition_notes }`。

**③ designTests（测试设计）**
- System：你是测试负责人，目标是把 Spec/Shards 转换成可执行、可判定的测试矩阵。
- User：Spec + Shards + 代码库测试栈摘要 + 输出 schema。
- 约束：必须覆盖单元、集成、E2E、回归、人工验收；每个必测项关联验收标准或风险；标注 required。
- 输出：Test Matrix（符合 [05 §2.6]）。

**④ reviewTests（测试审查）**
- System：你是严格测试审查者，默认不相信"自测通过"，必须基于测试证据判断。
- User：Test Matrix + Test Record + 失败日志 + coverage 摘要 + 相关 diff。
- 约束：结论 ∈ {TEST-PASS, TEST-MISSING, TEST-FAIL, TEST-FLAKY}；非 PASS 必须给出失败归因和修复/补测试指令。
- 输出：Test Review Record / Test Record 更新。

**⑤ review（单 Shard 实现审查）**
- System：你是严格审查者，默认挑错，给出可执行的修订指令。
- User：Shard（目标+验收）+ Implementation Record（diff 摘要+自测）+ 相关代码片段。
- 约束：逐条对照验收项；verdict ∈ {PASS, NEEDS-WORK, FAIL}；非 PASS 必须给 `fix_instructions`。
- 输出：Review Record（符合 [05 §2.5]）。

**⑥ reviewGlobal（整体复查）**
- System：你负责跨 Shard 一致性，关注单 Shard 审查看不到的整体问题。
- User：Spec + Test Matrix + 所有 Shard 的最终实现摘要 + Test Records + 关键接口/命名清单。
- 约束：查跨模块一致性、遗漏需求、回归、架构破坏、测试矩阵缺口。
- 输出：`{ verdict, issues[] }`（issues 触发回环时并入新 Spec）。

> 标注（annotate）可复用 review 输出，叠加标注维度生成 annotations（[05 §4]）。

## 3. ClaudeCodeAdapter · 实施

### 3.1 调用方式
- 以**非交互模式**启动 Claude Code，将 Shard（目标+自包含上下文）作为初始 prompt 注入。
- 工作目录指向目标代码仓库 `project_root`。
- 设置超时（默认 15 分钟）与最大轮次，避免失控。
- 进程结束后回收：改动文件清单、关键决策、自测结果、测试文件变更（Implementation Record）。

### 3.2 建议命令形态（示意，以实际 CLI 版本为准）
```
claude -p "<shard_prompt>" \
  --output-format json \
  --max-turns N \
  --permission-mode <受控> \
  --add-dir <project_root>
```
> Adapter 封装上述细节；关键是"单次任务、受控权限、结构化输出"。

## 4. SSO 与 Web Issue Intake

开发期不把飞书作为前置依赖，Issue 先由 Web 控制台提交。所有用户必须通过 Dofe SSO 登录。

### 4.1 登录源

| 服务 | 责任 |
|------|------|
| `sso.dofe.ai` | 唯一登录入口，负责用户登录态建立 |
| `api.sso.dofe.ai` | 用户信息、角色、权限校验与同步 |

NestJS `AuthModule` 通过 `SsoAdapter` 校验请求身份；Next.js 页面只向已登录用户开放 Issue 提交、审核和看板能力。

### 4.2 Web Issue 提交接口

```http
POST /api/issues
```

请求语义：

- 必须携带 Dofe SSO 登录态；
- 表单字段包含标题、目标仓库、需求正文、优先级、验收标准；
- 后端记录 `source_channel=web`、提交用户、原始表单 payload；
- 校验不通过时返回字段级错误，不进入 Codex 规划。

### 4.3 Web IssueIntake Schema

```yaml
---
id: intake-2026-0042-a
source_channel: web
source_kind: web_form
submitter:
  provider: dofe-sso
  user_id: user-001
  name: 张三
  roles: [developer]
message:
  create_time: 2026-06-19T14:22:01+08:00
raw_payload_ref: .loops/intakes/intake-2026-0042-a.raw.json
status: NORMALIZED      # RECEIVED | NEEDS-CLARIFICATION | NORMALIZED | REJECTED
---
```

## 5. FeishuAdapter · 后续 Issue 入口与通知

飞书在开发期后置。等 Web 提交、SSO、审核台、核心 Loop 跑通后，再把飞书接入为新的 `source_channel=feishu`。

### 5.1 接收事件

NestJS 暴露飞书事件回调接口：

```http
POST /api/integrations/feishu/events
```

处理流程：

1. 校验飞书事件请求（签名/加密/URL verification，按飞书开放平台当前配置实现）。
2. 解析事件类型：私聊消息、群聊 @、消息卡片交互、补充回复。
3. 抽取提交人、租户、会话、消息 id、文本、附件、原始 payload。
4. 写入 `IssueIntake`，并根据规则创建或更新 Issue。
5. 如信息不足，调用 `sendClarification()` 在飞书追问。

### 5.2 IssueIntake Schema（后续）

```yaml
---
id: intake-2026-0042-a
source_channel: feishu
source_kind: bot_direct_message   # bot_direct_message | group_mention | message_action | card_action
tenant_key: cli_xxx
submitter:
  provider: feishu
  open_id: ou_xxx
  union_id: on_xxx
  name: 张三
conversation:
  chat_id: oc_xxx
  chat_type: group
message:
  message_id: om_xxx
  root_id: om_root_xxx
  create_time: 2026-06-19T14:22:01+08:00
raw_payload_ref: .loops/intakes/intake-2026-0042-a.raw.json
status: NORMALIZED      # RECEIVED | NEEDS-CLARIFICATION | NORMALIZED | REJECTED
---
```

### 5.3 归一化规则

飞书消息进入 Codex 规划前必须满足：

- 有明确标题或可由正文生成标题；
- 有目标仓库或可根据群组/用户默认配置推断；
- 有可理解的需求正文；
- 有提交人身份与权限映射；
- 原始消息 payload 已落盘或入库，可追溯。

不满足时进入 `NEEDS-CLARIFICATION`，机器人追问。用户在同一飞书线程/会话回复后，系统把回复作为新的 `IssueIntake` 追加到同一个 Issue。

### 5.4 反向通知

系统应在以下节点向飞书通知提交人或审核群：

| 节点 | 通知内容 |
|------|----------|
| Issue 已接收 | Issue id、标题、状态、是否需要补充 |
| Spec 待审核 | Spec 摘要、审核入口、通过/打回按钮 |
| Loop 开始实施 | Shard 数、预计并发、目标仓库 |
| 阻塞/超限 | 阻塞原因、需要人处理的动作 |
| 收敛完成 | PR/提交摘要、终态标注链接 |

## 6. 上下文隔离设计（核心）

这是规避上下文压缩的工程落点：

| 设计点 | 做法 | 目的 |
|--------|------|------|
| **一 Shard 一进程** | 每个 Shard 启动全新 Claude Code 进程 | 进程结束即销毁上下文，无长上下文可供压缩 |
| **最小注入** | 只注入该 Shard 的自包含上下文 + 必要项目背景 | 不喂冗余历史 |
| **不跨 Shard 复用** | 不同 Shard 的进程互不可见彼此上下文 | 避免串扰、避免累积 |
| **全局状态外置** | 哪些做了/没做/整体一致性，存在 `.loops/`，由 Codex 纵览 | 不依赖任何实施进程的记忆 |
| **测试证据外置** | 测试命令、日志、覆盖率、失败归因保存为 Test Record | Codex 基于证据审查，而非相信口头自测 |
| **背景裁剪** | 项目背景按 Shard 相关性裁剪（如只给相关目录树/接口） | 控制 est_context |

> 验证手段：可在 Adapter 注入前后统计 token，确保 `est_context < budget`，超预算则拒绝启动并要求重拆。

## 7. 容错与重试

| 场景 | 策略 |
|------|------|
| Codex 输出不符合 schema | 解析失败重试（最多 N 次，附 schema 提示），仍失败则报错告警 |
| Claude Code 超时 | kill → 重试（默认 2 次）→ 转 BLOCKED 告警 |
| Claude Code 崩溃 / 非零退出 | 同上重试策略 |
| 测试命令失败 | 生成 Test Record，Codex 判断实现问题/测试问题/环境问题 |
| 必须测试缺失 | Shard 不得 DONE，回 Phase 4 要求补测试 |
| 网络/API 失败 | 指数退避重试，超限告警 |
| SSO 登录态无效 | 拒绝请求并引导重新登录 |
| SSO 用户无权限 | 拒绝提交/审核/介入，并记录审计事件 |
| Web Issue 表单信息不足 | 返回字段级错误或进入 `NEEDS-CLARIFICATION`，不启动 Codex |
| 飞书事件重复投递 | 按 `message_id/event_id` 幂等去重 |
| 飞书消息信息不足 | 进入 `NEEDS-CLARIFICATION`，追问补齐，不启动 Codex |
| 飞书用户未授权 | 拒绝创建 Issue，通知管理员或提交人 |
| 单 Shard 反复 NEEDS-WORK | 超"最大重做次数"升级 FAILED（[03 §7]） |
| 成本熔断 | token/调用超阈值 → 暂停 Loop + 告警 |

所有失败**落日志 + 落状态**，绝不静默。

## 8. 安全与权限

| 维度 | 要求 |
|------|------|
| **文件访问** | Claude Code 进程以白名单目录访问，禁止越界（`--add-dir` 受控） |
| **执行沙箱** | 建议在沙箱/容器或受控 permission mode 下运行，限制危险命令 |
| **密钥** | API key 等密钥不进入 Prompt、不进入日志；从环境/密钥管理读取 |
| **统一登录** | 只信任 `sso.dofe.ai` / `api.sso.dofe.ai`，禁止本地自建密码体系 |
| **权限校验** | 提交 Issue、审核 Spec、人工介入、运行 Runner 均需角色校验 |
| **飞书验签** | 所有飞书事件必须通过签名/加密校验，失败请求不落入 Issue 队列 |
| **用户映射** | 飞书用户必须映射到 Loops 用户和权限角色，禁止匿名高权限操作 |
| **原始 payload** | 原始 payload 可加密存储或脱敏展示，保留审计能力 |
| **审计** | 每次 agent 调用、每次文件改动可追溯 |
| **人工高危操作** | `loops take` 接管、删除等敏感动作需确认 |

## 9. 与 Git 的协作（已定方案 · 见 ADR-008）

按"commit-per-shard + 收敛 PR"落地：

- **commit-per-shard**：每个 Shard 实施完成（Claude Code 产出 + Codex 审查 PASS 转 `DONE`）即产生一次提交。
  - commit message 关联 `shard-id`，便于追溯与单独 revert；
  - Adapter 在 ClaudeCodeAdapter 完成实施后、状态落 `DONE` 时触发提交。
- **收敛 PR**：Loop 收敛（整体复查 PASS + 终态标注完成）后，聚合本 Loop 所有 shard commit，产出**一个 PR**。
  - PR 描述附**标注摘要**（各 Shard 的实现状态/审查结论/风险/已知限制）；
  - PR 由人最终合并——这是"人在环"在交付侧的体现。
- **分支策略**：每个 Loop 在独立分支/工作区推进（如 `loops/<issue-id>`），避免污染主干；并发实施时注意按 DAG 顺序提交，冲突按 [10 Q9] 处理。
