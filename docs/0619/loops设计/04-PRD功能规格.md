# 04 · PRD 功能规格

本章把 [03-工作流](03-工作流设计.md) 中的流程拆解为**可实施的功能模块**，每个模块给出职责、用户故事与验收标准。

---

## 模块全景

| 模块 | 名称 | 对应流程 | 一句话 |
|------|------|---------|--------|
| M1 | Issue Intake 与管理 | Phase 0/0.5 | Web 页面提交、归一化、队列、优先级、来源追溯 |
| M2 | 文档生成 | Phase 1 | Codex 把 Issue 变 Spec |
| M3 | 审核工作台 | Phase 2 | 人工门禁 |
| M4 | 拆解引擎 | Phase 3 | Spec → Shards（受预算约束） |
| M5 | 实施调度器 | Phase 4 | N×Claude Code 并发隔离实施 |
| M6 | 测试审查与标注引擎 | Phase 5/8 | Codex 测试设计 + 测试证据审查 + 标注 |
| M7 | 收敛与循环控制 | Phase 6/7 | 收敛判定 + 回环 |
| M8 | 可观测与日志 | 全程 | 状态、进度、成本 |
| M9 | 人工介入 | 全程 | 随时新开 agent / 接管 |

---

## M1 · Issue Intake 与管理

**职责**：开发期接收用户通过 Web 控制台提交的 Issue，归一化后进入待处理队列，管理优先级、生命周期与来源追溯；飞书/CLI/API 作为后续入口扩展。

- **US-1.1**：作为登录用户，我能在 Web 控制台提交 Issue，填写标题、目标仓库、需求正文、优先级与验收标准。
- **US-1.2**：作为用户，我能查看 Issue 队列与状态（OPEN / IN_LOOP / CLOSED / ARCHIVED）。
- **US-1.3**：作为用户，我能为 Issue 设置优先级（P0–P3）。
- **US-1.4**：作为系统，我必须记录每个 Issue 的提交用户、提交方式、原始表单 payload、附件与后续补充记录。
- **US-1.5**：作为系统，当表单缺少标题、目标仓库、验收标准或权限信息时，我能在页面提示补齐，禁止进入 Codex 规划。
- **US-1.6**：作为管理员，我能配置哪些 SSO 用户/角色可以提交 Issue，以及默认目标仓库/优先级。

**验收**：Issue 创建后生成唯一 `id` 并写入 `.loops/` 与数据库索引；每条 Issue 都有 `source_channel=web`、`submitter`、`raw_payload_ref`；信息不足时不会进入 Codex 规划，而是在页面提示补齐或进入 `NEEDS-CLARIFICATION`。

### M1.1 · Web 入口与 SSO 能力

| 能力 | 说明 | MVP |
|------|------|-----|
| SSO 登录 | 使用 `sso.dofe.ai` 作为唯一登录入口 | 必做 |
| 用户信息校验 | 使用 `api.sso.dofe.ai` 获取用户身份、角色、权限 | 必做 |
| Web Issue 表单 | 标题、目标仓库、需求正文、优先级、验收标准 | 必做 |
| 表单校验 | 缺少关键字段时不进入 Codex 规划 | 必做 |
| 来源追溯 | 记录登录用户、提交入口、原始表单 payload | 必做 |
| 飞书入口 | 后续接入机器人/消息卡片 | 后置 |

## M2 · 文档生成

**职责**：调用 Codex，把 Issue + 代码库现状转化为《产品文档》。

- **US-2.1**：作为系统，选取 OPEN 优先级最高的 Issue，调用 Codex 生成 Spec（含背景/范围/方案/验收/风险/拆解指引）。
- **US-2.2**：作为用户，我能看到生成过程与产物，并可要求 Codex 重生成。
- **US-2.3**：生成失败时，系统记录错误并重试，仍失败则告警。

**验收**：产物符合 [05 §2.2 Spec Schema]；生成过程留日志；失败有明确状态。

---

## M3 · 审核工作台（门禁）

**职责**：呈现 Spec 供人审核，支持通过/打回/拒绝。

- **US-3.1**：作为用户，我在 Next.js 审核台能阅读 Spec 并附批注。
- **US-3.2**：作为用户，我选择"通过"，Spec 进入拆解；选择"打回"，附批注回 M2 修订；选择"拒绝"，归档。
- **US-3.3**：作为系统，未经审核的 Spec 不得进入拆解（强制门禁）。
- **US-3.4**：作为提交人，我能在 Web 控制台看到"Spec 待审核"提醒；后续可扩展飞书通知。

**验收**：门禁状态可追溯；打回批注被传递给 Codex 并体现在 Spec v2。

---

## M4 · 拆解引擎

**职责**：调用 Codex，按上下文预算与拆解维度把 Spec 切成自包含的 Shard + 依赖 DAG。

- **US-4.1**：作为系统，读取已 APPROVED 的 Spec 与 `context_budget`，调用 Codex 拆解。
- **US-4.2**：作为系统，我对拆解结果做**质量自检**（自包含/覆盖完整/无重叠/不超预算），不通过则要求 Codex 重拆。
- **US-4.3**：作为用户，我能查看 Shard 列表、依赖图与预估工作量。

**验收**：每个 Shard 符合 [05 §2.3 Shard Schema]；`est_context < budget`；验收项合并覆盖 Spec 全部验收。

---

## M5 · 实施调度器

**职责**：按依赖 DAG 与并发上限，调度独立的 Claude Code 实例逐个实施 Shard。

- **US-5.1**：作为系统，我选取依赖已满足的就绪 Shard，在并发上限内启动 Claude Code 进程。
- **US-5.2**：作为系统，我保证每个进程**上下文隔离**（仅载入该 Shard + 必要背景）。
- **US-5.3**：作为系统，进程超时/崩溃时按重试策略处理，超限转 BLOCKED 并告警。
- **US-5.4**：作为系统，我回收 Implementation Record 并更新 Shard 状态。

**验收**：并发不超 `MAX_PARALLEL`；进程隔离可验证（无跨 Shard 串扰）；超时/重试有日志。

---

## M6 · 测试审查与标注引擎

**职责**：调用 Codex，对实施产出做单 Shard 审查、测试设计复核、测试证据判定与标注；收敛后做终态标注。

- **US-6.1**：作为系统，对每个 `IMPLEMENTED` Shard 调用 Codex 审查，产出 verdict + fix_instructions。
- **US-6.2**：作为系统，我要求 Codex 根据 Spec/Shards 生成并维护 `Test Matrix`，覆盖单元、集成、端到端、回归与人工验收。
- **US-6.3**：作为系统，我收集 Runner 执行测试的命令、日志、覆盖率、失败原因，形成 `Test Evidence`。
- **US-6.4**：作为系统，Codex 必须对测试证据做判定：`TEST-PASS` / `TEST-MISSING` / `TEST-FAIL` / `TEST-FLAKY`。
- **US-6.5**：作为系统，只有实现审查 PASS 且必须测试项 PASS 的 Shard 才能转 `DONE`。
- **US-6.6**：作为系统，收敛后对**所有文档**做终态标注刷新（无 unreviewed/partial/test-missing 遗漏）。
- **US-6.7**：作为用户，我能在标注看板看到每项的实现状态、测试状态、审查结论、风险、位置。

**验收**：审查结论符合 [05 §2.5]；测试记录符合 [05 §2.6]；标注符合 [05 §4]；终态无遗漏标注；任何必须测试缺失或失败都不得收敛。

---

## M7 · 收敛与循环控制

**职责**：判定收敛、驱动继续/回环、保护循环不失控。

- **US-7.1**：作为系统，我按 [03 §4 收敛判据] 判定是否收敛。
- **US-7.2**：作为系统，未收敛时回 Phase 4 继续推动（新 Round）。
- **US-7.3**：作为系统，整体复查 NEEDS-WORK 时触发回环（Spec 版本+1，回 Phase 2）。
- **US-7.4**：作为系统，我执行循环保护（最大回环次数、单 Shard 最大重做次数、成本熔断）。
- **US-7.5**：作为系统，超限或异常时暂停并向人告警。

**验收**：收敛判据四项（任务、测试、整体、标注）严格生效；回环有版本递增；保护机制触发即告警。

---

## M8 · 可观测与日志

**职责**：全程可见、可追溯、可重建。

- **US-8.1**：作为用户，我能看到当前 Loop 的阶段、Round、各 Shard 状态、并发数、累计成本。
- **US-8.2**：作为用户，我能查看每次 agent 调用的输入摘要、输出摘要、耗时、token。
- **US-8.3**：作为用户，我能查看标注看板（按文档/状态/风险筛选）。
- **US-8.4**：作为系统，我维护不可变日志，支持从日志重建状态。
- **US-8.5**：作为用户，我能从任一 Issue 追溯到原始 Web 表单、提交人、补充记录、审批记录和所有 agent 调用；后续飞书入口接入后可追溯原始飞书消息。

**验收**：进度面板实时反映状态；日志可查询；状态可重建。

---

## M9 · 人工介入

**职责**：让人随时介入，加速或修正。

- **US-9.1**：作为用户，我能 `loops exec <shard|desc>` 新开一个 Claude Code 实例。
- **US-9.2**：作为用户，我能 `loops review` / `loops plan` 新开一个 Codex 实例做审查/修订。
- **US-9.3**：作为用户，我能 `loops take <shard>` 接管为人工实施。
- **US-9.4**：作为用户，我能 `loops pause` / `loops resume` 暂停恢复整个 Loop。
- **US-9.5**：作为系统，所有人工介入同样写回状态与日志，标注来源 `HUMAN`。

**验收**：介入指令即时生效；介入产物并入 Loop 状态，不破坏收敛逻辑。

## 非功能需求

| 维度 | 要求 |
|------|------|
| **可靠性** | agent 崩溃/超时不丢失状态；可从日志重建。 |
| **可恢复性** | 任意时刻 kill 进程后，`loops resume` 能从断点继续。 |
| **测试可信** | 必须测试项失败、缺失或 flaky 时不得标记 DONE；Codex 需给出失败归因与下一步修复指令。 |
| **成本可控** | token/调用次数熔断；单 Loop 成本上限可配。 |
| **隔离/安全** | Claude Code 进程按白名单访问文件；沙箱执行；不泄露密钥。 |
| **可观测** | 全程日志；进度与成本面板。 |
| **可扩展** | agent 角色可替换（Codex/Claude Code 接口抽象）。 |
| **性能** | 调度开销可忽略；瓶颈在 agent 调用本身。 |
| **可移植** | 本地优先，跨平台（macOS/Linux/WSL）。 |
| **入口审计** | 外部入口必须保留原始 payload 引用、签名校验结果、用户映射结果，便于安全追溯。 |

## 产品化页面规格（Next.js）

| 页面 | 目标用户 | 核心内容 |
|------|----------|----------|
| `/issues` | 产品/工程负责人 | Issue 队列、来源、提交人、状态、优先级、目标仓库 |
| `/issues/new` | 登录用户 | 提交 Issue 表单、目标仓库选择、验收标准 |
| `/issues/[id]` | 提交人/审核人 | 原始需求、Web 来源、Spec 版本、Loop 进度、文档标注 |
| `/reviews` | 审核人 | 待审核 Spec、批注、通过/打回/拒绝 |
| `/loops/[id]` | 工程负责人 | Phase、Round、Shard DAG、运行中 agent、成本、阻塞项 |
| `/tests` | 工程/测试负责人 | Test Matrix、测试证据、失败归因、覆盖缺口 |
| `/annotations` | 审核/交付 | 按 Issue/Shard/风险/状态查看所有标注 |
| `/settings/auth` | 管理员 | SSO 配置、用户角色、权限规则 |
| `/settings/integrations/feishu` | 管理员 | 飞书应用配置、事件订阅状态、用户映射、权限规则（后置） |

## 后端服务规格（NestJS）

| 模块 | 责任 |
|------|------|
| `AuthModule` | 对接 `sso.dofe.ai` / `api.sso.dofe.ai`，维护登录态、用户、角色、权限 |
| `IssueModule` | Issue Intake、归一化、队列、状态机 |
| `SpecModule` | Codex 生成/修订产品文档 |
| `LoopModule` | Phase 0–8 状态机、收敛控制、回环 |
| `ShardModule` | 拆解结果、DAG、上下文预算校验 |
| `RunnerModule` | Claude Code CLI / Codex CLI 子进程调度、超时、重试 |
| `TestModule` | Test Matrix、测试命令编排、测试证据、失败归因 |
| `ReviewModule` | 单 Shard 审查、测试审查、整体复查、标注刷新 |
| `GitModule` | worktree、commit-per-shard、收敛 PR |
| `AuditModule` | 不可变日志、事件回放、来源追溯 |
