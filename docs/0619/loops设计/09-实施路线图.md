# 09 · 实施路线图

本章给出 Loops 从 **MVP 到完整版**的分阶段路线。核心策略：先用 `npx create-dofe-ai` 搭好 NestJS + Next.js 产品骨架，再跑通"SSO 登录 → Web 提交 Issue → Web 审核 → Codex/Claude Code 闭环"的最小产品链路。飞书入口后置。

---

## v1 当前裁剪说明（2026-06-19）

本章原路线保留为长期目标；当前 v1 worktree 统一采用更小的主链路，避免 SSO/飞书/真实 PR 等外部依赖阻塞核心闭环。

```text
无登录 Web 提交 Issue
  -> Issue / Intake / LoopState 存储到数据库索引
  -> 同步写入并继续维护 .loops 文档真相源
  -> 使用现有 Loop 流程完成开发
  -> 整体复查与终态标注
  -> Issue CLOSED
```

v1 明确不做：真实登录/SSO、权限角色、飞书入口、飞书消息卡片、真实远端 PR 打开、多 worker 生产级并发、生产级告警。上述能力进入后续版本，不作为 v1 任务依赖或验收项。

---

## 里程碑总览

```
M0 产品脚手架  →  M1 WebIssue+审核台+串行闭环  →  M2 并发+隔离+完整标注
                                              │
              M4 可观测+介入+Git+熔断  ←  M3 回环+整体复查+终态标注
```

| 里程碑 | 主题 | 核心验证 |
|--------|------|---------|
| **M0** | 产品脚手架与内核骨架 | DofeAI monorepo / NestJS / Next.js / Adapter / 存储可跑 |
| **M1** | Web Issue + 审核台 + 最小闭环（串行） | **核心价值**：页面提交 + 人审 + 双 agent + 测试证据 + 规避压缩 + 收敛 |
| **M2** | 并发 + 隔离 + 完整测试标注 | 多 Claude Code 并发、上下文隔离、测试矩阵、标注体系 |
| **M3** | 回环 + 整体复查 + 全局回归 + 终态标注 | 循环收敛完整、全局测试收敛、终态标注 |
| **M4** | 可观测 + 介入 + Git + 熔断 | 生产可用：面板、随时介入、成本安全 |

## M0 · 产品脚手架与内核骨架

**目标**：在固定项目目录中搭好产品骨架，能接收请求、调通 agent、落状态。

**创建位置**：

```text
/Users/techwu/Documents/codes/dofe.ai/vibecoding.dofe.ai
```

**脚手架命令**：

```bash
cd /Users/techwu/Documents/codes/dofe.ai
npx create-dofe-ai vibecoding.dofe.ai
```

**交付物**：
- Next.js Web 骨架：SSO 登录接入、Issue 提交页、Issue 队列、Spec 审核台、Loop 状态页占位。
- NestJS API 骨架：`AuthModule`、`IssueModule`、`LoopModule`、`RunnerModule`、`TestModule`、`ReviewModule`。
- `loops` CLI 骨架（`resume`/`status`/`doctor` 等运维命令）。
- CodexAdapter / ClaudeCodeAdapter 骨架（能各自成功调用一次 agent）。
- SsoAdapter 骨架（能校验登录态 / mock user）。
- `.loops/` 目录结构 + `state.json` + `log.jsonl`。
- 配置加载（`config.yaml` + 后台设置页占位）。

**验收**：
- `apps/web` 和 `apps/api` 能本地启动。
- 手动能用 Adapter 各调用一次 Codex / Claude Code 并解析输出。
- 能通过 Web 表单或 mock user 创建一条 `IssueIntake`。
- 日志与状态正确写入。

**风险**：agent CLI 非交互调用方式与版本相关 → 缓解：Adapter 封装 + 实现时核对版本。

## M1 · MVP：Web Issue + 审核台 + 最小闭环（串行）

> **这是验证产品价值的关键里程碑——先串行跑通，不求并发。**

> v1 当前裁剪：M1 的“SSO 登录用户”改为“无登录 Web 用户或 mock submitter”；仍必须记录提交入口与原始 payload。数据库用于查询索引和状态聚合，`.loops` 继续作为文档真相源。

**目标**：对一个从 Web 页面提交的真实 Issue，端到端跑完：SSO 登录 → Web Issue 表单 → Issue Intake → Codex 生成 Spec + Test Matrix → Next.js 人工审核 → Codex 拆解 → **串行**逐个 Claude Code 实施 → Runner 执行测试 → Codex 测试审查 + 实现审查 → 全部 DONE → 终态标注 → CLOSED。

**范围裁剪（MVP 简化）**：
- v1 不接入真实 SSO；使用无登录表单或 mock submitter 完成 Issue Intake。
- 串行实施（`max_parallel=1`），先不做并发。
- 暂不做回环（整体复查 FAIL 仅告警，不自动重写）。
- 标注做基础维度（impl_status / test_status / verdict / location）。
- 测试矩阵先做基础维度：unit/integration 必须项，E2E 可先标记为 manual 或 deferred。
- **Git**：启用 commit-per-shard（每个 Shard 审查 PASS 转 `DONE` 时提交，关联 `shard-id`）。
- **真实远端 PR**：后置；v1 可保留收敛 PR 描述或本地记录，不要求打开 provider PR。
- Web 先支持 Issue 新建、Spec 审核、Issue 详情、Loop 进度四页。
- 飞书入口、消息卡片审批、附件解析后置。

**验收（核心价值验证清单）**：
- [ ] 无登录 Web 用户或 mock submitter 能在 Web 页面创建 Issue，并记录提交入口、原始表单 payload、提交时间和提交人占位字段。
- [ ] Issue / Intake / LoopState 能写入数据库索引，并与 `.loops` 文档证据保持一致；冲突时以 `.loops` 为准。
- [ ] 信息不足时页面能提示补齐，未补齐前不进入 Codex 规划。
- [ ] Codex 产出的 Spec 经人审核后可拆解。
- [ ] Codex 能生成 Test Matrix，并把必须测试项关联到 Shard。
- [ ] 每个 Shard 由独立 Claude Code 进程实施，`est_context < budget`（**隔离有效**）。
- [ ] Runner 能执行测试命令并生成 Test Record。
- [ ] Codex 能审查测试证据与实现，并给出 PASS/NEEDS-WORK。
- [ ] 必须测试缺失或失败时，Shard 不得进入 DONE。
- [ ] 全部 DONE 后完成终态标注，Issue CLOSED。
- [ ] kill 后 `loops resume` 能恢复。

**意义**：若 M1 跑通，即证明"双 agent 分工 + 拆解规避压缩 + 测试证据 + 文档标注闭环"的核心假设成立。

## M2 · 并发 + 上下文隔离 + 完整测试标注

**目标**：把"串行"升级为"按 DAG 并发"，强化隔离与标注。

**交付物**：
- Scheduler：依赖 DAG + `max_parallel` 并发调度。
- 上下文隔离强化：token 预算校验（超预算拒绝启动 + 要求重拆）。
- 完整测试矩阵：unit / integration / e2e / regression / manual。
- 完整标注体系（impl_status / test_status / verdict / coverage / location / risk / notes）+ 标注看板数据。
- 失败/超时/重试完整链路。
- 通知能力完善：Web 站内提醒；飞书反向通知后置。

**验收**：
- 多 Shard 并发不超上限，依赖被正确尊重。
- 超预算 Shard 被拦截。
- 必须测试项缺失/失败时阻止 Shard DONE。
- 标注看板可按状态/风险筛选。

---

## M3 · 回环 + 整体复查 + 全局回归 + 终态标注

**目标**：闭合"循环收敛"的最后一环。

**交付物**：
- `reviewGlobal`（Codex 整体复查）。
- 全局回归：合并所有 Shard 后执行回归测试、关键 E2E、构建、lint。
- 回环机制：整体 NEEDS-WORK → Spec 版本+1 → 回 Phase 2。
- 循环保护：最大回环次数、单 Shard 最大重做、超限告警。
- 终态标注刷新（无 unreviewed/partial/test-missing 遗漏）。

**验收**：
- 能演示一次完整回环并二次收敛。
- 能演示一次测试失败触发修复或回环。
- 保护机制触发即告警，不死循环。

---

## M4 · 可观测 + 人工介入 + Git + 成本熔断

**目标**：达到"日常可用、可控成本、随时介入"。

**交付物**：
- 可观测：Next.js 进度面板、成本统计、日志查询、通知记录。
- 人工介入：`exec` / `review` / `plan` / `take` / `pause` / `resume`。
- Git 集成（可选）：commit-per-shard、收敛产出 PR + 标注摘要。
- 成本熔断：token/调用阈值暂停 + 告警。

**验收**：
- 面板实时反映 Loop 状态与成本。
- 介入指令即时生效且落账。
- 成本超限自动暂停。

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| agent 非交互调用不稳/版本变动 | 调用失败 | Adapter 封装 + 输出 schema 校验 + 重试 |
| SSO 对接不稳定 | 无法登录/提交 | v1 不接入真实 SSO，使用无登录表单或 mock submitter；真实 `sso.dofe.ai` 后置 |
| 飞书事件回调本地不可达 | 后续飞书入口受阻 | 飞书集成后置；开发期不阻塞核心闭环 |
| 拆解质量差（不自包含/超预算） | 实施跑偏/压缩 | 拆解自检 + 预算硬约束 + 人工审核门禁 |
| 测试覆盖不足导致假收敛 | 缺陷漏出 | Codex Test Matrix + 必须测试项阻断 DONE + 全局回归 |
| 并发文件冲突 | 改动互相覆盖 | DAG 依赖 + 文件白名单 + 必要时串行 |
| 成本失控 | 费用/时间 | 熔断 + 上限 + 告警 |
| 无限回环 | 死循环 | 最大回环次数 + 超限人工介入 |
| 单 agent 既定 provider 不可用 | 流程中断 | Adapter 可替换为其他 agent |
