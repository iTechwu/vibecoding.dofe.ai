# 08 · 数据存储设计

本章定义 Loops 的**状态管理、实体模型、目录布局与日志规范**。`.loops/` 是整个系统的**唯一真相源**——可读、可 git、可重建。

---

## v1 当前裁剪说明（2026-06-19）

v1 会引入数据库持久化以支撑 Web 查询、列表筛选与状态聚合，但不改变本章的真相源原则：

- `.loops/` 仍是文档真相源，Issue、Intake、Spec、Shard、Test Record、Review Record、Annotation 与 `log.jsonl` 都必须能作为最终证据被读取和归档。
- 数据库是查询索引与状态聚合来源，不是唯一真相源；数据库记录必须可由 `.loops/` 文档与不可变日志重建。
- v1 数据库范围聚焦 `Issue / IssueIntake / LoopState` 及必要的 Shard/Test/Review/Annotation 索引；SSO 用户、外部身份、飞书身份映射保留为后续扩展。
- 发生冲突时，以 `.loops/` 文档和 `log.jsonl` 为准，并由 doctor/一致性检查暴露差异。

---

## 1. 设计原则

| 原则 | 说明 |
|------|------|
| **文件优先** | 状态与产物以文件（JSON/YAML/Markdown）为主，人可直接读、可 git |
| **数据库辅助索引** | 查询/聚合走 SQLite 或 PostgreSQL，但核心文档仍可由文件重建 |
| **不可变日志** | `log.jsonl` 追加写，记录一切，可据此重建状态 |
| **版本保留** | Spec/Shard 历史版本归档不删，支持回环与可追溯 |
| **幂等可恢复** | kill 后 `loops resume` 能从 `.loops/` 完整恢复 Loop 状态 |

## 3. 实体模型（ER）

```
Issue 1──N IssueIntake
Issue 1──1 Spec 1──N TaskShard
                  │
                  │ depends_on (DAG, N:N)
                  │
              TaskShard 1──N Run(round)
                              │
                              │ 1──1 ImplementationRecord
                              │ 1──N TestRecord
                              │ 1──1 ReviewRecord

TestMatrix 1──N TestRecord
Annotation N──1 (Issue | Spec | TaskShard)   # 贯穿各层
LogEntry (append-only，关联任意实体)
```

### 3.1 实体字段（与 [05 文档协议] 一致）

- **IssueIntake**：`id, issue_id, source_channel, source_kind, submitter, conversation, message, raw_payload_ref, status`
- **Issue**：`id, title, status, priority, created, source_channel, submitter_id, target_repo, body`
- **Spec**：`id, issue, version, status, context_budget, approved_by, body`
- **TaskShard**：`id, spec, title, status, priority, depends_on[], est_context, est_effort, acceptance[], files_hint[]`
- **Run**：`shard, round, agent, status, files_changed[], duration_sec, tokens, summary, known_issues[]`
- **TestMatrix**：`id, issue, spec, status, required_tests[], regression_scope[]`
- **TestRecord**：`id, scope, round, status, commands[], coverage, failed_tests[], fix_instructions[]`
- **ReviewRecord**：`shard|global, reviewer, round, verdict, issues[], fix_instructions[]`
- **Annotation**：`target, annotator, round, impl_status, test_status, verdict, coverage, location[], risk, notes`

## 5. log.jsonl（不可变日志）

每行一条事件，追加写，**禁止修改/删除**。

```jsonc
// 示例条目
{"ts":"2026-06-19T14:00:00Z","type":"AGENT_CALL","agent":"codex","op":"decompose","loop":"issue-2026-0042","tokens_in":5200,"tokens_out":8800,"ms":18300,"ok":true}
{"ts":"2026-06-19T14:01:12Z","type":"SHARD_STATE","loop":"issue-2026-0042","shard":"shard-2","from":"TODO","to":"IN_PROGRESS"}
{"ts":"2026-06-19T14:12:55Z","type":"SHARD_STATE","loop":"issue-2026-0042","shard":"shard-2","from":"IN_PROGRESS","to":"IMPLEMENTED"}
{"ts":"2026-06-19T14:13:30Z","type":"TEST_RUN","loop":"issue-2026-0042","shard":"shard-2","status":"TEST-FAIL","commands":["pnpm test scheduler"]}
{"ts":"2026-06-19T14:14:10Z","type":"TEST_REVIEW","loop":"issue-2026-0042","shard":"shard-2","verdict":"TEST-FAIL"}
{"ts":"2026-06-19T14:15:00Z","type":"REVIEW","loop":"issue-2026-0042","shard":"shard-2","verdict":"NEEDS-WORK"}
{"ts":"2026-06-19T14:30:00Z","type":"RELOOP","loop":"issue-2026-0042","reason":"global review NEEDS-WORK","new_spec_version":"v2"}
{"ts":"2026-06-19T15:00:00Z","type":"HUMAN_INTERVENE","loop":"issue-2026-0042","action":"exec","target":"shard-4"}
```

事件类型：`AGENT_CALL | SHARD_STATE | SPEC_STATE | ISSUE_STATE | TEST_MATRIX | TEST_RUN | TEST_REVIEW | REVIEW | ANNOTATE | RELOOP | CONVERGE | HUMAN_INTERVENE | COST_TRIP | ERROR`。

补充外部入口事件类型：

```jsonc
{"ts":"2026-06-19T13:55:00+08:00","type":"INTAKE_RECEIVED","source":"web","intake":"intake-2026-0042-a","submitter":"user-001"}
{"ts":"2026-06-19T13:55:03+08:00","type":"ISSUE_NORMALIZED","issue":"issue-2026-0042","intake":"intake-2026-0042-a","target_repo":"blog-app"}
{"ts":"2026-06-19T13:56:00+08:00","type":"CLARIFICATION_REQUESTED","issue":"issue-2026-0042","channel":"web","question":"请补充目标仓库"}
{"ts":"2026-06-19T13:58:12+08:00","type":"CLARIFICATION_RECEIVED","issue":"issue-2026-0042","intake":"intake-2026-0042-b"}
```

外部入口相关事件类型：`INTAKE_RECEIVED | INTAKE_REJECTED | ISSUE_NORMALIZED | CLARIFICATION_REQUESTED | CLARIFICATION_RECEIVED | NOTIFY_SENT`。

## 6. 状态重建（Recovery）

`loops resume` 的恢复逻辑：
1. 读 `state.json` 得各 Loop 当前 phase / round。
2. 读各 Shard 最新状态，校验与日志一致。
3. 对 `IN_PROGRESS` 但无对应活跃进程的 Shard → 视为中断，按重试策略处理（转 `TODO` 或重试）。
4. 对 phase 落在"实施中"的 Loop → 恢复 Scheduler 继续调度。
5. 若 `state.json` 损坏 → 从 `log.jsonl` 重放重建。

> 保证：任意时刻 kill，状态不丢，可恢复。

## 8. 数据库索引建议

`.loops/` 仍是文档真相源，但产品化页面需要快速查询。MVP 可用 SQLite，团队部署建议 PostgreSQL。

> v1 说明：当前 v1 优先落 PostgreSQL/DB Service 索引能力以服务 Web 页面和 Loop 状态查询；`users`、`external_identities` 等真实身份表不作为 v1 前置项，只有在无登录/mock submitter 需要稳定引用时才保留最小字段。

| 表 | 主要字段 | 说明 |
|----|----------|------|
| `users` | `id, display_name, email, role` | Loops 用户 |
| `external_identities` | `user_id, provider, external_user_id, tenant_key` | SSO/飞书等外部身份映射 |
| `issue_intakes` | `id, issue_id, source_channel, source_kind, submitter_id, raw_payload_ref, status` | 所有入口记录 |
| `issues` | `id, title, status, priority, target_repo, submitter_id, source_channel` | Issue 查询索引 |
| `loops` | `id, issue_id, phase, round, reloop_count, paused` | Loop 顶层状态 |
| `shards` | `id, loop_id, status, priority, est_context, depends_on` | Shard 看板 |
| `test_matrices` | `id, issue_id, spec_version, status, required_count, passed_count` | 测试矩阵 |
| `test_records` | `id, shard_id, round, status, commands, coverage, failed_count` | 测试证据 |
| `agent_runs` | `id, shard_id, agent, op, status, duration_sec, tokens` | agent 调用与成本 |
| `annotations` | `id, target_type, target_id, verdict, test_status, coverage, risk` | 标注看板 |
| `audit_events` | `id, ts, type, actor_id, entity_ref, payload_ref` | 不可变事件索引 |

数据库记录可以从 `.loops/` 与 `log.jsonl` 重建；若出现冲突，以 `.loops/` 文档和不可变日志为准。
