# 03 · 工作流设计（核心）

本章定义 Loops 的心脏——**完整 Loop 的阶段、状态机、收敛判据、并发与介入机制**。所有其他文档都围绕这里定义的流程展开。

---

## v1 当前裁剪说明（2026-06-19）

本章保留长期目标中的 SSO、飞书、CLI/API 多入口、并发调度与真实 PR 语义；但当前 v1 统一按以下边界实施：

- v1 主链路：**无登录 Web 提交 Issue → Issue / Intake / LoopState 写入数据库索引并同步 `.loops` → 使用现有 Loop 流程完成开发 → 整体复查与终态标注 → Issue CLOSED**。
- v1 不把登录、SSO、权限角色、飞书入口、飞书通知、真实远端 PR 打开作为前置条件或验收项。
- Phase 0 / 0.5 在 v1 中使用匿名或 mock submitter；仍记录提交入口、原始表单 payload、目标仓库、提交时间和补齐状态。
- `.loops/` 仍是文档真相源和可重建证据；数据库只作为查询索引与状态聚合来源。若 DB 与 `.loops` 冲突，以 `.loops` 文档和不可变日志为准。
- 长期设计中的 SSO、飞书、多 worker 并发、生产级告警与真实 PR 进入后续版本，不阻塞 v1 worktree。

---

## 1. 总体循环图

一个 Loop 由 9 个阶段（Phase 0–8）组成，构成一个闭环：

```
┌─────────────────────────────────────────────────────────────────────┐
│                          Loops · 一个完整 Loop                        │
│                                                                       │
│  Phase 0  提交 Issue                                                    │
│      │                                                                │
│      ▼                                                                │
│  Phase 1  Codex 生成《产品文档》  ──┐                                   │
│      │                              │                                  │
│      ▼                              ▼                                  │
│  Phase 2  【门禁】人工审核文档 ──▶ 不通过 ──▶ 回 Phase 1 修订           │
│      │  通过                                                          │
│      ▼                                                                │
│  Phase 3  Codex 拆解文档 → N 个 Task Shard（受上下文预算约束）           │
│      │                                                                │
│      ▼                                                                │
│  Phase 4  并发实施：N × Claude Code CLI（上下文隔离，逐个实现）          │
│      │                                                                │
│      ▼                                                                │
│  Phase 5  Codex 测试审查 + 标注（PASS / NEEDS-WORK / FAIL）            │
│      │                                                                │
│      ▼                                                                │
│  Phase 6  收敛判定 ◀────────────────────────────────────┐             │
│      │   ┌────────────────────────┬──────────────┐      │             │
│      │   │ 有剩余/未通过 Shard      │ 全部 DONE     │      │             │
│      │   ▼                        ▼              │      │             │
│      │  Phase 4' 继续/重试实施    Phase 7 整体复查 │      │             │
│      │   (新轮次 Round++)             │           │      │             │
│      │   └─────────────┬──────────────┘           │      │             │
│      │                 │                          │      │             │
│      │                 ▼                          │      │             │
│      │        ┌───────────────┐                   │      │             │
│      │        │ 整体复查 PASS? │── 否 ─────────────┼──────┘ (回环)     │
│      │        └───────┬───────┘                   │                    │
│      │           是   │                           │                    │
│      │                ▼                           │                    │
│      │        Phase 8  Codex 终态标注所有文档 ──▶ 交付（Loop 结束）      │
│      │                                            │                    │
│      │   回环（Re-loop）：Codex 重写《产品文档》，回到 Phase 2            │
│      └────────────────────────────────────────────┘                    │
│                                                                       │
│  ※ 任意 Phase 期间，人可随时新开 Claude Code / Codex 介入（见 §6）       │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. 各阶段定义

### Phase 0 · 提交 Issue
- **角色**：人。
- **动作**：开发期通过 Web 控制台提交一个 Issue（自然语言需求），进入待处理队列；飞书消息、CLI、API 作为后续入口扩展。
- **v1 裁剪**：不要求登录/SSO；Web 表单可匿名或使用 mock submitter 提交。飞书、CLI、API 入口均为后置扩展，不作为 v1 主链路。
- **Web 优先入口**：
  - 用户通过 Dofe SSO 登录后，在页面填写标题、目标仓库、需求正文、优先级、验收标准；
  - 系统生成 `Issue Intake`，记录登录用户、提交入口、表单原始 payload、提交时间；
  - 若信息不足，页面直接提示补齐，暂不进入 Codex 规划。
- **产出**：`Issue` 记录（含原始描述、提交时间、优先级、来源渠道、提交人、原始表单）。
- **状态变更**：`Issue: OPEN`。

### Phase 0.5 · Issue 归一化与确认
- **角色**：系统 + 人。
- **动作**：把来自 Web 表单的输入统一成标准 Issue；后续再兼容飞书/CLI/API。
- **v1 裁剪**：只归一化 Web 表单输入；提交人身份使用匿名或 mock submitter 字段，不接入真实身份映射。
- **归一化内容**：
  - 标题、正文、目标代码仓库、业务背景、期望结果；
  - 提交人身份、提交渠道、原始表单、附件/图片/链接；
  - 优先级、可见范围、是否需要人工确认。
- **确认策略**：
  - 简单明确的 Issue 可直接进入 `OPEN`；
  - 缺少目标仓库、验收标准或权限不明时，停留在 Web 表单校验或进入 `NEEDS-CLARIFICATION`；
  - 用户补充后合并到同一个 Issue，保留多轮 intake 记录。
- **状态变更**：`Issue: INTAKE → OPEN / NEEDS-CLARIFICATION / REJECTED`。

### Phase 1 · 生成产品文档
- **角色**：Codex。
- **动作**：读取 Issue，结合目标代码库现状，生成《产品文档》（需求、范围、方案、验收标准、风险）与初始测试策略（测试层级、关键用例、回归范围）。详见 [05-文档与任务模型](05-文档与任务模型.md)。
- **产出**：`Product Spec`（v 待审核）。
- **状态变更**：`Spec: DRAFT`。

### Phase 2 · 【门禁】人工审核
- **角色**：人。
- **动作**：审阅《产品文档》。可：通过 / 打回修订（附批注）/ 拒绝。
- **门禁语义**：这是 Loop 中**唯一默认强制的人工门禁**——保障方向正确后才进入实施。
- **流转**：
  - 通过 → Phase 3；
  - 打回 → 回 Phase 1（Codex 按批注修订）；
  - 拒绝 → Loop 终止（归档）。
- **状态变更**：`Spec: APPROVED` / `REVISION_REQUESTED` / `REJECTED`。

### Phase 3 · 拆解文档为 Task Shard
- **角色**：Codex。
- **动作**：按**上下文预算**与拆解策略，把《产品文档》切成 N 个自包含的 Task Shard。每个 Shard 含：目标、上下文、依赖、验收标准、测试要求、预估工作量。详见 [05-文档与任务模型 §3](05-文档与任务模型.md)。
- **关键约束**：任一 Shard 预估上下文 > 预算 → 必须进一步拆分。
- **产出**：`Task Shard[]`，构成一棵（可能有依赖的）任务树/DAG。
- **状态变更**：各 Shard 初始化为 `TODO`。

### Phase 4 · 并发实施
- **角色**：Claude Code CLI × N。
- **动作**：对每个可执行 Shard（依赖已满足），启动一个**独立、隔离**的 Claude Code 进程实施。详见 [07-集成与接口](07-集成与接口.md)。
- **隔离**：进程只载入该 Shard 上下文 + 必要项目背景；进程结束上下文销毁。
- **产出**：`Implementation Record`（改动文件、关键决策、自测结果、已新增/已修改测试）。
- **状态变更**：`Shard: TODO → IN_PROGRESS → IMPLEMENTED`（或失败 `BLOCKED`）。
- **并发**：受全局并发上限约束，按依赖 DAG 调度（见 §5）。

### Phase 5 · 测试审查 + 标注
- **角色**：Codex。
- **动作**：对每个 `IMPLEMENTED` 的 Shard：
  1. **测试设计复核**：核对 Claude Code 是否补齐该 Shard 要求的单元/集成/端到端测试；缺失则 `NEEDS-WORK`。
  2. **测试执行判定**：读取 Runner 的测试命令、日志、覆盖率、失败信息，形成 `Test Record`。
  3. **实现审查**：核对实现是否满足该 Shard 的目标与验收标准，结论 `PASS` / `NEEDS-WORK` / `FAIL`。
  4. **标注**：在文档与任务上标注实现状态、测试状态、审查结论、风险、覆盖度（见 [05-文档与任务模型 §4](05-文档与任务模型.md)）。
- **流转**：
  - `PASS` 且测试通过 → `Shard: DONE`；
  - 测试缺失/失败或审查 `NEEDS-WORK` → 回 Phase 4，附修订指令（该 Shard 重新实现或补测试）；
  - `FAIL` → 标记问题，进入 Phase 6 收敛判定（可能触发回环）。
- **状态变更**：`Shard: IMPLEMENTED → DONE / NEEDS-WORK / FAILED`。

### Phase 6 · 收敛判定
- **角色**：编排引擎（自动）。
- **动作**：检查是否所有 Shard = `DONE`。
  - **否（仍有 TODO / NEEDS-WORK / FAILED）** → 回 Phase 4 继续推动（进入新 Round，`Round++`）；若问题属于"方案层面"而非"实现层面"，则触发**回环**到 Phase 2。
  - **是（全部 DONE）** → 进入 Phase 7。

### Phase 7 · 整体复查
- **角色**：Codex。
- **动作**：在所有 Shard 完成后，做**跨 Shard 的整体一致性复查与全局测试复核**——这是单 Shard 审查覆盖不到的：
  - 跨模块接口一致性、命名/风格统一；
  - 是否有遗漏的需求、未实现的边界；
  - 整体架构是否被破坏、是否引入回归。
  - 全局测试矩阵是否完整，关键集成/E2E/回归测试是否通过。
- **结论**：
  - `PASS` → Phase 8；
  - `NEEDS-WORK` → 触发**回环**：Codex 重写《产品文档》（把新发现的问题纳入），回到 Phase 2，开启新一轮 Loop。
- **回环次数保护**：设最大回环次数上限，避免无限循环（见 §7）。

### Phase 8 · 终态标注
- **角色**：Codex。
- **动作**：收敛达成后，对所有文档（Issue、Spec、Shards、实施记录、测试记录、审查记录）做**最终的准确标注**——确保每一项都标注了：最终实现状态、测试状态、所在位置、审查结论、风险与已知限制。
- **产出**：闭环交付物 + 完整标注的文档集。
- **状态变更**：`Issue: CLOSED`，Loop 结束。

## 3. 状态机

### 3.1 Issue 状态
```
OPEN ──▶ IN_LOOP ──▶ CLOSED
                  ╲
                   ╲──▶ ARCHIVED（被拒绝/废弃）
```

### 3.2 Product Spec 状态
```
DRAFT ──▶ APPROVED ──▶ (进入拆解)
  ▲          │
  │          ├──▶ REVISION_REQUESTED ──▶ DRAFT（修订）
  │          └──▶ REJECTED（归档）
  └─────────────── (回环时 Re-loop: 新版本 v++)
```

### 3.3 Task Shard 状态（核心）
```
                ┌──────────────────────────────┐
                ▼                               │
TODO ──▶ IN_PROGRESS ──▶ IMPLEMENTED ──▶ DONE   │
  │           │              │                  │
  │           │              ├──▶ NEEDS-WORK ───┘ (重做, 回 IN_PROGRESS)
  │           │              │
  │           ▼              ▼
  │        TIMEOUT        FAILED ──▶ (进入收敛判定 / 可能回环)
  │           │
  └───────────┘ (重试)
```

**状态语义：**
| 状态 | 含义 |
|------|------|
| `TODO` | 已拆解，待实施（依赖未满足也停留在此）。 |
| `IN_PROGRESS` | Claude Code 正在实施。 |
| `IMPLEMENTED` | 实施完成，待 Codex 审查。 |
| `DONE` | 实施完成、必须测试通过且 Codex 审查 PASS。收敛的计数单元。 |
| `NEEDS-WORK` | 审查或测试认为不达标，附修订/补测试指令重做。 |
| `FAILED` | 实现存在严重问题，需收敛判定决定回环或废弃。 |
| `BLOCKED` | 依赖未满足或资源不可用，暂停。 |
| `TIMEOUT` | 超时，进入重试策略。 |

## 4. 收敛判据（严格定义）

**Loop 收敛 ⟺ 以下三式同时成立：**

1. **任务收敛**：`∀ Shard ∈ Loop, Shard.status = DONE`
2. **测试收敛**：`∀ RequiredTest ∈ TestMatrix, RequiredTest.status = PASS`
3. **整体收敛**：`Codex.整体复查 = PASS`
4. **标注收敛**：`所有文档的必填标注项均完整`

任一不满足，Loop 继续或回环。收敛是唯一的正常出口。

## 5. 并发、依赖与调度

### 5.1 依赖 DAG
Shard 之间可能存在依赖（如"先实现数据层，再实现接口层"）。拆解时 Codex 标注每个 Shard 的 `depends_on`，构成 DAG。

### 5.2 调度规则
- 只有 `所有 depends_on 均为 DONE` 的 Shard 才可进入 `IN_PROGRESS`。
- 全局并发上限 `MAX_PARALLEL`（默认 4，可配），同时运行的 Claude Code 进程不超过此数。
- 进程结束立即释放槽位，调度下一个就绪 Shard。

### 5.3 资源与成本控制
- 每个 Claude Code 进程有**超时上限**（默认 15 分钟）与**最大重试次数**（默认 2）。
- 累计 token / 调用次数超阈值时，暂停并向人告警（成本熔断）。

## 6. 随时介入机制

> "期间可以随时启动新的 claude code cli / codex 进行进一步的代码实施。"

这是 Loops 的一等公民特性，而非附属功能。介入方式：

| 介入类型 | 触发 | 作用 |
|---------|------|------|
| **新开实施** | 人手动 `loops exec <shard-or-desc>` | 启动一个额外 Claude Code 实例，可针对某 Shard 或自由描述的任务。 |
| **新开审查/修订** | 人手动 `loops review` 或 `loops plan` | 启动一个额外 Codex 实例做审查或修订文档。 |
| **接管** | 人手动 `loops take <shard>` | 把某 Shard 标记为人工实施，跳过 Claude Code。 |
| **暂停 / 恢复** | `loops pause` / `loops resume` | 暂停或恢复整个 Loop。 |

**介入的落账**：所有人工介入同样写回 `.loops/` 状态与日志，标注来源为 `HUMAN`，保持可追溯。介入不破坏 Loop 的收敛逻辑——它只是加速或修正某个 Shard/文档的状态。

## 7. 循环保护与异常处理

| 风险 | 保护机制 |
|------|---------|
| 无限回环 | **最大回环次数上限**（默认 3），超限则暂停并请求人工介入。 |
| 单 Shard 反复 NEEDS-WORK | **单 Shard 最大重做次数**（默认 3），超限升级为 FAILED，触发收敛判定。 |
| 测试长期失败 | Codex 必须判断是实现问题、测试问题还是方案问题；实现问题回 Phase 4，测试问题补测试，方案问题触发回环。 |
| Agent 崩溃 / 超时 | 自动重试，超重试上限转 BLOCKED 并告警。 |
| 成本失控 | token/调用次数熔断，暂停并告警。 |
| 状态损坏 | `.loops/` 为唯一真相源，支持从日志重建状态。 |

## 8. 一次 Loop 的端到端示例（叙事）

> v1 示例请按“无登录 Web 提交 Issue”理解；下方 SSO 叙事保留为长期目标示例，不作为 v1 验收要求。

1. **人**通过 Dofe SSO 登录 Web 控制台，提交 Issue："给博客系统加评论的草稿与定时发布功能。"
1. **系统**记录登录用户、入口、目标仓库、原始表单 payload；若缺少目标仓库，页面提示补齐，形成标准 Issue。
2. **Codex** 生成《产品文档》：范围、数据模型改动、API、UI、验收、风险。
3. **人**审核 → 打回一处数据模型，Codex 修订 → 通过。
4. **Codex** 拆解为 5 个 Shard：①DB 迁移 ②草稿 API ③定时任务 ④前端草稿 UI ⑤前端定时 UI；标注依赖：④⑤ 依赖 ②。
5. **Phase 4**：①②③ 并发实施（3 个 Claude Code 实例）；②完成后 ④⑤ 加入并发。
6. **Phase 5**：Codex 审查测试证据 → ③定时任务缺少时区边界测试且实现没处理好（NEEDS-WORK），其余 DONE；③补实现与测试后 PASS。
7. **Phase 6**：全部 DONE → Phase 7。
8. **Phase 7**：整体复查发现 ④⑤ 用了不一致的 loading 风格，且全局 E2E 未覆盖定时发布失败态 → NEEDS-WORK → **回环**，Codex 把"统一 loading 规范 + E2E 失败态覆盖"写进文档 v2 → 回 Phase 2。
9. 第二轮 Loop：重拆、实施新增的小 Shard、审查通过、整体复查 PASS。
10. **Phase 8**：Codex 对所有文档终态标注（实现状态、文件位置、审查结论、已知限制）→ `Issue: CLOSED` → 交付。
