# 05 · 文档与任务模型

本章定义 Loops 中**所有文档的结构、拆解规则与标注体系**。这是"文档即真相"原则的落地——拆解决定 Claude Code 能否规避上下文压缩，标注决定整个 Loop 是否可审计、可收敛。

---

## 1. 文档层级与流转

Loops 的文档构成一条从需求到交付的链：

```
Issue ──▶ Product Spec ──▶ Task Shards ──▶ Implementation Records
                         │                    │
                         ▼                    ▼
                    Test Matrix          Test Records
                                              │
                                              ▼
                                       Review Records
                                              │
                                              ▼
                                       Annotations（贯穿所有层，多轮演进）
```

| 层级 | 文档 | 创建者 | 作用 |
|------|------|--------|------|
| L0 | Issue | 人 | 原始需求 |
| L0 | Issue Intake | 系统 | 飞书/Web/CLI 等外部入口的原始提交记录 |
| L1 | Product Spec | Codex | 审核后的完整方案（唯一进入实施的入口） |
| L1 | Test Matrix | Codex | 从 Spec/Shards 派生出的测试矩阵与回归范围 |
| L2 | Task Shard | Codex | 自包含的最小实施单元 |
| L3 | Implementation Record | Claude Code | 单 Shard 的实施过程与产物 |
| L3 | Test Record | Runner + Codex | 测试命令、日志、覆盖率、失败归因与结论 |
| L3 | Review Record | Codex | 单 Shard / 整体的审查结论 |
| L4 | Annotation | Codex | 贯穿各层的结构化状态标注 |

## 2. 文档协议（Schema）

所有文档以 Markdown 为主体 + YAML frontmatter 承载结构化元数据，便于人读、机器解析。

### 2.1 Issue
```yaml
---
id: issue-2026-0042
title: 给博客系统加评论的草稿与定时发布
status: OPEN          # INTAKE | NEEDS-CLARIFICATION | OPEN | IN_LOOP | CLOSED | ARCHIVED
priority: P1
created: 2026-06-19
source_channel: web
submitter_id: user-001
target_repo: /Users/techwu/workspace/blog-app
---
（自然语言需求正文）
```

### 2.1.1 Issue Intake
```yaml
---
id: intake-2026-0042-a
issue: issue-2026-0042
source_channel: web          # web | feishu | cli | api
source_kind: web_form        # web_form | bot_direct_message | group_mention | message_action | cli
submitter:
  provider: dofe-sso
  user_id: user-001
  name: 张三
message:
  create_time: 2026-06-19T14:22:01+08:00
raw_payload_ref: .loops/intakes/intake-2026-0042-a.raw.json
status: NORMALIZED           # RECEIVED | NEEDS-CLARIFICATION | NORMALIZED | REJECTED
---
## 原始消息摘要
## 归一化结果
## 追问与补充
```

### 2.2 Product Spec
```yaml
---
id: spec-0042
issue: issue-2026-0042
version: v1            # 回环时 +1
status: DRAFT          # DRAFT | APPROVED | REVISION_REQUESTED | REJECTED
created: 2026-06-19
approved_by: human
context_budget: 24000  # 本 Spec 拆解时采用的上下文预算(token)
---
## 背景与目标
## 范围（含明确的不做项）
## 方案设计
## 验收标准（可勾选清单）
## 测试策略（单元 / 集成 / E2E / 回归 / 人工验收）
## 风险与依赖
## 拆解指引（给 Codex 拆解阶段的提示）
```

### 2.3 Task Shard（核心）
```yaml
---
id: shard-0042-3
spec: spec-0042 / v1
title: 实现定时发布调度任务
status: TODO           # 见 03 §3.3 状态机
priority: P1
depends_on: [shard-0042-2]   # 依赖的 Shard
est_context: 9000      # 预估上下文 token（必须 < context_budget）
est_effort: M          # S | M | L
acceptance:            # 该 Shard 的验收清单
  - 能在指定时间自动发布草稿
  - 失败有重试与告警
test_requirements:
  unit:
    - 覆盖调度时间计算和重试策略
  integration:
    - 覆盖数据库状态从草稿到发布的转换
  e2e:
    - 用户创建定时草稿后到点自动发布
files_hint:            # 涉及文件提示（可空）
  - src/scheduler/*.ts
---
## 目标
## 上下文（自包含：必要背景、约束、接口契约）
## 实施要求
## 验收标准
## 测试要求
```

### 2.4 Implementation Record
```yaml
---
shard: shard-0042-3
agent: claude-code
round: 1
status: IMPLEMENTED
files_changed:
  - src/scheduler/publish.ts (+82 -3)
  - tests/scheduler.test.ts (+44)
duration_sec: 412
tokens: 31200
tests_changed:
  - tests/scheduler.test.ts
---
## 实施摘要
## 关键决策
## 自测结果
## 已知问题 / TODO
```

### 2.5 Review Record
```yaml
---
shard: shard-0042-3       # 或 scope: global 表示整体复查
reviewer: codex
round: 1
verdict: NEEDS-WORK     # PASS | NEEDS-WORK | FAIL
issues:
  - severity: major
    desc: 定时任务未处理时区，跨天会错发
fix_instructions:       # 给 Claude Code 的修订指令
  - 使用 UTC 存储，展示层转本地时区
---
## 审查意见
```

### 2.6 Test Matrix
```yaml
---
id: test-matrix-0042
issue: issue-2026-0042
spec: spec-0042 / v1
owner: codex
status: DRAFT          # DRAFT | ACTIVE | PASS | NEEDS-WORK
required_tests:
  - id: test-0042-unit-1
    shard: shard-0042-3
    level: unit        # unit | integration | e2e | regression | manual
    title: 定时发布时间计算覆盖 UTC 与本地时区
    command: pnpm test scheduler
    required: true
  - id: test-0042-e2e-1
    shard: global
    level: e2e
    title: 用户创建定时草稿后到点自动发布
    command: pnpm e2e scheduled-publish
    required: true
---
## 测试矩阵说明
## 必须测试项
## 回归范围
## 暂不自动化的人工验收项
```

### 2.7 Test Record
```yaml
---
id: test-record-0042-3-r1
scope: shard-0042-3      # 或 global
round: 1
runner: loops-runner
reviewer: codex
status: TEST-FAIL        # TEST-PASS | TEST-MISSING | TEST-FAIL | TEST-FLAKY | SKIPPED
commands:
  - pnpm test scheduler
  - pnpm lint
coverage:
  lines: 82.1
  branches: 76.4
failed_tests:
  - name: handles cross-day scheduled publish
    reason: expected UTC conversion but got local timestamp
fix_instructions:
  - 使用 UTC 存储调度时间，并补充跨天测试
---
## 测试执行摘要
## 失败归因
## Codex 测试审查结论
```

## 3. 拆解规则（Decomposition）

拆解是 Loops 的第一公民。它的目标是：**把 Spec 切成若干自包含、不超预算、依赖清晰的 Shard。**

### 3.1 上下文预算（Context Budget）

- **定义**：单个 Shard 允许的最大上下文 token。
- **默认值**：`24000` token（保守，远离压缩阈值；可按模型上限调整，建议 ≤ 模型上限的 40%）。
- **硬约束**：`est_context ≥ budget` 的 Shard **必须进一步拆分**，否则不准进入 Phase 4。
- **预算构成**：Shard 正文 + 注入的项目背景 + 预期对话轮次开销。

### 3.2 拆解维度

Codex 按以下维度（可组合）切分，优先保证**自包含**：

| 维度 | 适用 | 示例 |
|------|------|------|
| **按模块/子系统** | 模块边界清晰 | 数据层 / API 层 / UI 层 |
| **按文件/目录** | 改动可按文件聚合 | 改 `scheduler/*` 一组 |
| **按功能点** | 功能彼此独立 | 草稿功能 / 定时功能 |
| **按依赖层（自底向上）** | 有明确分层 | DB迁移 → Model → Service → API → UI |

### 3.3 自包含性要求（每个 Shard 必须满足）

1. **目标明确**：能用一句话说清这个 Shard 要交付什么。
2. **上下文自足**：不依赖"看过整份 Spec"才能干活——必要背景、接口契约、约束都内联进 Shard。
3. **验收可判**：有可勾选的验收清单，Codex 能据此判 PASS/FAIL。
4. **测试可判**：有明确的测试要求，Codex 能据此判断测试缺失/失败/通过。
5. **依赖显式**：`depends_on` 列全。
6. **不超预算**：`est_context < budget`。

> 自包含是规避压缩的关键：Claude Code 拿到 Shard 就能干活，不需要长历史。

### 3.4 拆解的输出

拆解阶段产出：
- Shard 列表（含上述 frontmatter）；
- 依赖 DAG（`depends_on` 关系图）；
- 拆解说明（为什么这么切，给可追溯用）。

### 3.5 拆解质量自检（Codex 在拆解后自检）

- [ ] 每个 Shard 都自包含？
- [ ] 所有 Shard 合起来是否覆盖了 Spec 全部验收项？（**覆盖完整性**）
- [ ] 所有 Shard 合起来是否覆盖了 Test Matrix 的必须测试项？
- [ ] 是否有 Shard 重叠/冲突？
- [ ] 是否有 Shard 超预算未拆？

未通过自检则重新拆解，避免实施阶段才发现遗漏。

## 4. 标注体系（Annotation）

> 用户在原始构想中反复强调"标注"——审查后标注、终态标注。标注是 Loop 可审计、可收敛的基石。

### 4.1 标注的定位

标注 ≠ 审查。**审查是判断（PASS/FAIL），标注是把判断与状态固化进文档**，让任何人在任何时候都能看到"这一项现在是什么状态、结论是什么、风险在哪"。

### 4.2 标注维度

每条标注覆盖以下维度：

| 维度 | 取值 | 说明 |
|------|------|------|
| `impl_status` | `not-started` / `in-progress` / `done` / `failed` / `skipped` | 实现状态 |
| `test_status` | `not-run` / `pass` / `missing` / `fail` / `flaky` / `skipped` | 测试状态 |
| `verdict` | `pass` / `needs-work` / `fail` / `unreviewed` | 审查结论 |
| `coverage` | `none` / `partial` / `full` | 相对其验收项的覆盖度 |
| `location` | 文件路径 / 行号 / 模块 | 实现落点（便于回溯） |
| `risk` | `low` / `medium` / `high` | 风险等级 |
| `notes` | 自由文本 | 已知限制、待办、说明 |

### 4.3 标注 Schema
```yaml
---
target: shard-0042-3          # 或 spec-0042 / issue-0042 / 全局
annotator: codex
round: 2
impl_status: done
test_status: pass
verdict: pass
coverage: full
location:
  - src/scheduler/publish.ts:45-120
risk: low
notes: 已处理 UTC 时区；未覆盖数据库不可用时的降级（见已知限制）
---
```

### 4.4 标注的落位

- **文档级标注**：写在文档 frontmatter（结构化、可查询）。
- **行内标注**：在 Spec/Shard 正文对应条目旁，用标注标记（如 `[✅ done]` `[⚠ needs-work]`）便于人快速浏览。
- **全局标注看板**：`.loops/annotations.yaml` 汇总所有标注，供可观测面板渲染。

### 4.5 标注的演进（多轮）

- 标注**按轮次（round）追加**而非覆盖，保留历史轨迹（v1 fail → v2 pass 可追溯）。
- 每次审查、每次回环后，Codex 更新对应标注。
- **终态标注（Phase 8）**：收敛后，Codex 对**所有文档**做一次最终、完整、准确的标注刷新，确保无 `unreviewed`、无 `partial`、无 `test-missing` 遗漏——这是收敛判据的来源。

## 5. 版本与可追溯

| 机制 | 说明 |
|------|------|
| **Spec 版本** | 每次回环 `version +1`，旧版本归档不删。 |
| **Shard 版本** | NEEDS-WORK 重做时，保留历次 Implementation/Review Record。 |
| **测试版本** | Test Matrix 随 Spec 版本演进；Test Record 按 round 保留。 |
| **标注按轮次** | 标注历史可查。 |
| **不可变日志** | `.loops/log.jsonl` 追加写，记录每次 agent 调用与状态变更，可据此重建状态。 |

> 所有产物都在 `.loops/`，构成完整的"为什么这么做、做到哪了、结论是什么"的证据链。存储细节见 [08-数据存储设计](08-数据存储设计.md)。
