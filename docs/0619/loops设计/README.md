# Loops — 自主编码循环引擎

> **一句话定位**：Loops 是一个以 **Codex 为大脑、Claude Code 为双手**的自主编码循环系统——它把一个 Issue 自动转化为经过拆解、实施、审查、标注、闭环交付的代码实现。

---

## 30 秒读懂 Loops

你提一个 Issue。Codex 把它变成产品文档；你审一下；Codex 再把文档**拆成多个小块**；每个小块交给一个独立的 Claude Code CLI 去实现（**天然规避上下文压缩**）；实现完 Codex 逐个审查并标注；没做完就继续，做完了 Codex 做整体复查；还有问题就**重写文档回到循环**，直到全部收敛；最后 Codex 对所有文档做准确标注。

**整个产品 = 一个不断收敛的循环（Loop）。** 这也是产品名的由来。

```
Issue → Codex 生成文档 → 人工审核 → Codex 拆解 → N×Claude Code 实施
   ↑                                                        ↓
   └── [仍有问题] Codex 重写文档 ← Codex 整体复查 ← Codex 审查+标注 ← 剩余继续
                                                          ↓
                                            [全部收敛] Codex 终态标注 → 交付
```

---

## 它解决什么问题

| 痛点 | Loops 的解法 |
|------|------------|
| Claude Code 跑长任务会触发**上下文压缩**，质量退化、遗忘上下文 | 用 Codex 把大任务**拆成小上下文**，每个 Claude Code 实例只啃一块 |
| 单 agent 一条龙，**既写又审**，缺乏交叉校验 | **双 agent 分工**：Codex 专司规划/审查/标注，Claude Code 专司实施 |
| AI 写完没人管，**文档与代码脱节** | Codex 在每一轮**审查并标注文档**，文档即真相 |
| 任务大、跑不完、容易半途而废 | **循环收敛机制**，剩余项持续推动，直到全部闭环 |

---

## 核心理念

1. **小上下文优于大上下文** —— 拆解是第一公民，规避压缩是硬约束。
2. **分工优于全能** —— Codex（规划/审查）与 Claude Code（实施）角色分离，交叉校验。
3. **循环优于一次性** —— Plan → Do → Check → Act，直到收敛。
4. **文档即真相** —— 每个产物都有状态标注，可追溯、可审计。
5. **人随时在环** —— 关键门禁由人把关，且可随时新开 cli / codex 介入。

## 关键决策（已确认）

以下决策已与产品方确认，作为后续文档与实现的基础：

| # | 决策 | 详见 |
|---|------|------|
| D1 | **大脑 = OpenAI Codex CLI**（`codex` 命令）；双手 = Anthropic Claude Code（非交互模式） | ADR-001 / ADR-009 |
| D2 | **产品形态：Web 控制台 + 后端编排服务优先**——Next.js 负责提交/审核/看板，NestJS 负责编排与 agent 调度，CLI 仅作为本地执行/运维入口 | ADR-007 |
| D3 | **Git 集成：commit-per-shard + 收敛 PR**——每 Shard 完成即提交、收敛后聚合为 PR 附标注摘要 | ADR-008 |
| D4 | **开发期 Issue 入口：Web 页面优先**——先在控制台提交 Issue，记录登录用户、入口、原始表单；飞书作为后续集成 | ADR-010 |
| D5 | **工程脚手架：`npx create-dofe-ai` + NestJS + Next.js**——后续项目目录固定为 `/Users/techwu/Documents/codes/dofe.ai/vibecoding.dofe.ai` | ADR-011 |
| D6 | **登录源：Dofe SSO**——统一使用 `sso.dofe.ai` 与 `api.sso.dofe.ai` 作为唯一身份来源 | ADR-013 |

其余仍为**假设/待定**（如并发上限、成本阈值、回环时增量补拆策略），见 [10-决策与开放问题](10-决策与开放问题.md)。

> 文档集为 v0.1 草案。并发上限、成本策略、飞书后续集成范围、部署形态等待定项可在 [10](10-决策与开放问题.md) 中调整。
