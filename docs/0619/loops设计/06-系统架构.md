# 06 · 系统架构

本章把 PRD 的 9 个功能模块落到**分层架构与组件**上，给出端到端数据流、技术选型与部署形态。产品化基线已调整为：**Dofe SSO + Next.js 控制台 + NestJS 编排服务 + Runner 执行层**。开发期 Issue 入口先做 Web 页面提交，飞书作为后续集成。

---

## 1. 架构总览

Loops 采用 **交互层 + NestJS 编排层 + Runner 执行层 + 能力适配层 + 存储层** 的五层结构。核心思想：Web 负责登录、输入与审核，NestJS 负责 SSO 校验、状态机与调度，Runner 隔离执行 Codex / Claude Code，agent 仍通过 Adapter 解耦。

```
Next.js / CLI / 飞书(后续)
        │
        ▼
NestJS API + Orchestrator
  ├─ Auth / SSO
  ├─ Web Issue Intake / Issue Queue
  ├─ Spec Generator / Decomposer
  ├─ Scheduler / Convergence Controller
  ├─ Test Planner / Reviewer / Annotator
  └─ Observer / Audit / Cost Guard
        │
        ▼
Runner Worker
  ├─ git worktree / branch
  ├─ Codex CLI calls
  ├─ Claude Code CLI calls
  ├─ tests / lint / build
  └─ diff / logs / records collection
        │
        ├────────▶ Adapters: Codex / Claude Code / SSO / Git / Feishu(后续)
        │
        ▼
Storage
  ├─ .loops/ 文档真相源
  ├─ PostgreSQL or SQLite 查询索引
  ├─ log.jsonl 不可变事件
  └─ runs / annotations / artifacts
```

---

## 2. 分层职责

### 2.1 交互层
- **Next.js Web 控制台**：Dofe SSO 登录、Issue 提交、Issue 队列、Spec 审核、Loop 看板、Shard DAG、标注看板、集成配置。
- **飞书入口（后续）**：支持机器人私聊、群聊 @、消息卡片审核、状态反向通知。
- **CLI**：保留为本地执行和运维入口，如 `loops resume`、`loops exec`、`loops doctor`，不再是唯一产品界面。

### 2.2 NestJS API + 编排层（Orchestrator）
Loops 的服务端大脑。实现 PRD 的 M1–M8 逻辑，不直接耦合 agent 细节，而是通过 Runner 与适配层。

- **Auth / SSO**：对接 `sso.dofe.ai` 与 `api.sso.dofe.ai`，校验登录态、同步用户身份与角色。
- **Web Issue Intake**：接收 Web 表单、校验必填项、记录提交用户与原始 payload。
- **Loop 控制器**：驱动 Phase 0–8 状态机、收敛判定、回环、循环保护。
- **拆解引擎**：调用 Codex 拆解 + 质量自检。
- **实施调度器**：依赖 DAG + 并发上限调度 Claude Code。
- **测试、审查 & 标注引擎**：调用 Codex 生成测试矩阵、审查测试证据、审查实现、写标注、终态标注。

### 2.3 Runner 执行层
Runner 负责所有长耗时、高风险、强本地状态的动作：

- 创建 git worktree / branch；
- 启动 Codex CLI、Claude Code CLI；
- 执行测试、lint、构建；
- 回收 diff、日志、Implementation Record、Test Record；
- 按超时、重试、并发上限隔离运行。

NestJS 可以通过 BullMQ/Redis、数据库 job 表或本机队列派发 Runner Job。MVP 可先单进程内置 Runner，M2 后拆为独立 worker。

### 2.4 能力适配层（Adapters）
把"调 agent"这件事抽象成统一接口，解耦具体厂商：
- **CodexAdapter**：封装 6 类调用（生成/拆解/测试设计/测试审查/实现审查/标注），见 [07](07-集成与接口.md)。
- **ClaudeCodeAdapter**：封装 headless 启动、上下文注入、超时重试、产物回收。
- **SsoAdapter**：封装登录态校验、用户信息查询、角色/权限同步。
- **FeishuAdapter（后续）**：封装事件验签、用户查询、消息发送、卡片交互。
- **GitAdapter**：封装 worktree、commit-per-shard、PR 创建。

> 适配层是可替换性的关键：换 agent 只换 Adapter，编排层不动。

### 2.5 存储层
`.loops/` 为文档真相源；数据库为产品查询、权限、飞书来源追溯、任务队列提供索引。见 [08](08-数据存储设计.md)。

## 4. 端到端数据流（一次 Loop）

```
1. 人 → Dofe SSO 登录 → Next.js Issue 表单 → WebIssueIntake：写入 Issue Intake
2. IssueQueue：归一化、表单校验、权限校验 → Issue (OPEN)
3. Loop控制器：取最高优先 Issue → SpecGenerator → CodexAdapter.plan() → Spec(DRAFT)
4. ReviewWorkbench：人审核 → APPROVED
5. Decomposer → CodexAdapter.decompose(Spec, budget) → Shards[] + DAG
   └─ 自检不通过 → 重拆
6. Scheduler：就绪 Shard → Runner → ClaudeCodeAdapter.run(shard)（隔离进程）
   └─ 回收 Implementation Record → Shard: IMPLEMENTED
7. Runner 执行测试 → Test Record
8. Reviewer → CodexAdapter.reviewTests(test_record) + review(shard, impl) → verdict
   └─ 测试 PASS 且审查 PASS → DONE；NEEDS-WORK → 回 5；FAIL → 进收敛判定
   └─ Annotator 写标注
9. ConvergenceController：
   ├─ 未全部 DONE → 回 5（Round++）
   └─ 全部 DONE → CodexAdapter.reviewGlobal() →
        ├─ PASS → Annotator 终态标注 → Issue: CLOSED
        └─ NEEDS-WORK → Spec v++ → 回 3（回环）
10. 全程：Observer 写 log.jsonl、更新看板、成本熔断检查；后续可向飞书反向通知关键状态
```

## 5. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 编排与 agent 解耦 | Adapter 模式 | agent 可替换；编排逻辑稳定 |
| 产品形态 | Next.js + NestJS | SSO 登录、Web Issue、审核台、看板、编排 API 都需要服务化 |
| 状态存储 | `.loops/` 文件 + 数据库索引 | 可读、可 git、可重建；数据库供产品查询 |
| 进程模型 | 每 Shard 一个独立 agent 进程 | 上下文隔离，规避压缩 |
| 并发模型 | DAG 调度 + 并发上限 | 尊重依赖、控制成本 |
| 失败语义 | 不静默吞错，全部落日志+状态 | 可追溯、可恢复 |
| 人在环 | 仅 Spec 审核为强制门禁 | 平衡自治与可控 |
| Issue 入口 | 开发期 Web 页面提交，GitHub Issue 可选同步，飞书后续集成 | 先降低开发复杂度，快速跑通核心闭环 |

## 6. 技术选型（建议）

| 维度 | 建议 | 理由 |
|------|------|------|
| 脚手架 | `npx create-dofe-ai vibecoding.dofe.ai` | 已确认使用 DofeAI monorepo 脚手架，npm 当前 latest 为 `0.1.27` |
| 后端 | **NestJS + TypeScript** | 模块化清晰，适合 API、Webhook、队列、权限、编排服务 |
| 前端 | **Next.js + TypeScript** | 审核台、看板、设置页、服务端渲染/全栈路由成熟 |
| CLI | commander / yargs | 运维、本地 runner、doctor、resume |
| 状态存储 | `.loops/` 文件 + PostgreSQL/SQLite 索引 | 文档可读可 git，产品查询走数据库 |
| 队列 | BullMQ/Redis 或数据库 job 表 | M1 可用数据库 job 表，M2 后引入 BullMQ |
| 进程管理 | Node.js `child_process` / `execa` / tmux 可选 | 启停 agent、超时控制、日志捕获 |
| 日志 | 追加式 JSONL + 数据库事件表 | 不可变、可回放、可看板查询 |
| 认证集成 | `sso.dofe.ai` + `api.sso.dofe.ai` | 唯一登录源、用户信息、角色权限 |
| 飞书集成 | Feishu Open Platform Webhook / Bot | 后续 Issue 输入、追问、审批通知 |
| 配置 | `.loops/config.yaml` + 后台设置页 | 预算、并发、超时、熔断阈值、飞书集成 |

> 选型已按用户方向固定为 NestJS + Next.js；Agent 与队列实现仍可通过 Adapter/Runner 演进。

## 7. 部署形态

**本地产品化优先**：Loops 作为本地/内网 Web 服务运行，`.loops/` 与目标代码仓库并列，Next.js/NestJS 提供控制台与 API。后续创建项目目录固定为：

```text
/Users/techwu/Documents/codes/dofe.ai/vibecoding.dofe.ai
```

建议 monorepo 布局：

```text
vibecoding.dofe.ai/
├── apps/
│   ├── web/                 # Next.js 控制台
│   ├── api/                 # NestJS API + Orchestrator
│   └── worker/              # Runner，可先与 api 合并
├── packages/
│   ├── core/                # Loop 状态机、实体、文档协议
│   ├── adapters/            # Codex/Claude/SSO/Git/Feishu adapters
│   ├── storage/             # .loops + DB repository
│   └── ui/                  # 共享 UI 组件
└── docs/
```

目标代码仓库侧典型布局：

```
my-project/              ← 被实施的目标仓库
├── src/...
└── ...
loops-design/            （可选）Loops 自身工程
my-project.loops/ 或 .loops/   ← Loops 工作目录（状态/文档/日志）
```

- 单机多 Loop：每个目标仓库一个 `.loops/`。
- Web 控制台作为本地或内网服务提供审核与可观测 UI。
- 不强制公网云化；飞书 Webhook 如需公网回调，可通过内网穿透、企业网关或部署到可信服务器。
- 团队协作时数据库可切换 PostgreSQL，`.loops/` 仍作为每个目标仓库的文档真相源。
