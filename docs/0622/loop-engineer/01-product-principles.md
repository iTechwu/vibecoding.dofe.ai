# 01 · 产品原则

## 目标用户

### 业务/产品负责人

他们关心的是需求是否被正确理解、交付是否满足验收标准、风险是否可控。不应该要求他们理解 shard DAG、Test Record 或 Review Record 的写入格式。

### 工程负责人

他们关心的是执行是否可追溯、测试是否充分、代码是否可审阅、失败是否能恢复。他们可以查看 shard 和证据，但不应该被迫手动填充每个环节。

### 平台/运维管理员

他们关心的是 runtime、workspace、权限、成本、队列和异常恢复。他们需要诊断入口，但不应该让普通用户承担这些操作。

## 核心原则

### 1. Intent in, evidence out

用户输入意图，系统输出证据。中间的计划、拆解、实现、测试和审阅由引擎负责。

用户应该说：

- “修复登录后跳转异常”
- “优化 Loops detail 页，让我不用手动填 shards”
- “给这个 issue 跑完整实现和回归”

用户不应该被要求说：

- “请先接管 shard-1”
- “我手动填写 Implementation Record”
- “我点击 Run Shard Tests 后再点击 Review Shard”

### 2. Human approval is a gate, not a workflow engine

人工批准只应该出现在判断意图和风险的地方：

- Spec 是否准确；
- 是否接受某个高风险变更；
- 成本/权限/回环上限触发后是否继续；
- 最终交付是否满足业务期望。

人工不应该负责补齐系统本应自动采集的工程证据。

### 3. Shards are internal, evidence is user-visible

Shard 是引擎为了控制上下文、并发、依赖和审阅边界而产生的内部单元。用户可以看到 shard 进度，但不应该管理 shard 生命周期。

推荐呈现：

- shard 标题、状态、依赖数量；
- 当前自动化步骤；
- implementation/test/review 证据数量；
- 最新测试和审阅结论；
- 对应验收标准。

避免呈现：

- 手动接管按钮；
- 手动填写 implementation/test/review 表单；
- 需要用户理解内部状态机才能继续的多个阶段按钮。

### 4. One primary action per state

每个页面状态只给用户一个主要动作：

- 没有 Spec：继续推进 Loop；
- Spec 草稿：批准或要求修改；
- Spec 已批准：继续推进 Loop；
- 执行中断：继续推进 Loop 会自动恢复；
- 全局 review 未通过：启动 re-loop 或查看阻断原因；
- 已完成：查看证据。

细粒度动作可以保留在 CLI、管理员工具或调试模式中，但不作为普通用户的默认入口。

### 5. Explain stoppage, not internals

当引擎停下时，页面应该回答三个问题：

1. 为什么停下；
2. 谁需要做什么；
3. 做完后如何继续。

不合格文案：

> No runnable shard is available.

合格文案：

> 当前 Spec 正在等待人工审阅。请在规格区域批准或要求修改；批准后引擎会自动拆解并继续实现。

## 体验反模式

| 反模式                        | 问题                   | 替代方案                                   |
| ----------------------------- | ---------------------- | ------------------------------------------ |
| Shard 手填表单                | 把引擎责任转嫁给用户   | 自动写证据，UI 只展示                      |
| 多个阶段按钮并列              | 用户需要理解内部状态机 | 一个 `Continue Loop` 主按钮                |
| 运行失败只显示技术错误        | 用户不知道下一步       | 异常卡片说明原因、责任人、恢复动作         |
| 把 runtime 细节暴露给普通用户 | 增加认知负担           | 普通用户看“引擎可用/不可用”，管理员看诊断  |
| 旧轮证据污染当前轮            | 用户误判当前状态       | re-loop 后默认展示当前 round，旧轮归档可查 |

## 产品指标

| 指标                 | 目标                                                                                   |
| -------------------- | -------------------------------------------------------------------------------------- |
| 新 issue 创建耗时    | 普通用户 1 分钟内完成                                                                  |
| 人工必需动作数量     | 正常 happy path 只有 Spec 审阅                                                         |
| Shard 手动表单使用率 | 默认用户路径为 0                                                                       |
| 自动恢复成功率       | `IN_PROGRESS` / `TIMEOUT` 可恢复场景接近 100%                                          |
| 停止原因可理解度     | 用户能从页面文案直接知道下一步                                                         |
| 证据完整率           | finalize 前 implementation/test/review/global review/convergence evidence 全部 present |
