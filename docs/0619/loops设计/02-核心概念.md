# 02 · 核心概念与角色模型

理解 Loops 需要先建立四件事：**两个 agent 如何分工**、**为什么要拆解（上下文压缩问题）**、**拆解不能解决什么**、**什么是"循环收敛"**。本章逐一讲清，并给出贯穿全文档的术语表。

---

## 1. 角色模型：大脑与双手

Loops 的世界里只有三个角色：**一个大脑、一双手、一个人**。

```
        ┌──────────────────────────────────────────────┐
        │                  人 (Human)                   │
        │   提交 Issue · 审核文档 · 最终决策 · 随时介入   │
        └───────────────┬──────────────────────┬───────┘
                        │ 下达 / 门禁            │ 介入
                        ▼                      ▼
        ┌──────────────────────┐     ┌──────────────────────┐
        │   Codex · 大脑        │     │  Claude Code · 双手   │
        │ ──────────────────    │     │ ──────────────────    │
        │ • 生成产品文档         │────▶│ • 逐个实施 Task Shard │
        │ • 拆解文档为小任务     │     │ • 修改代码 / 写测试    │
│ • 审查实施产出         │◀────│ • 返回实施记录         │
│ • 设计/审查测试方案     │     │ • 执行测试 / 修复失败   │
│ • 标注文档与状态       │     │ • 单一上下文，不复用    │
        └──────────────────────┘     └──────────────────────┘
```

### 1.1 Codex —— 大脑（规划 / 拆解 / 审查 / 标注）

Codex 不直接写业务代码，它做五件"高阶"的事：

| 职责 | 产出 |
|------|------|
| **规划 Plan** | 把 Issue 转成《产品文档》(Product Spec) |
| **拆解 Decompose** | 把文档切成若干**自包含的小任务 (Task Shard)** |
| **测试设计 Test Plan** | 为 Spec/Shards 生成测试矩阵、验收用例、回归清单 |
| **审查 Review** | 审查 Claude Code 的实施产出，给出结论 |
| **标注 Annotate** | 在文档/任务上标注状态、风险、覆盖度 |

> 设计意图：Codex 的上下文是"文档 + 全局视图 + 测试证据"，它需要纵览全局，因此把它放在大脑与测试负责人位置；它不陷入大段实现细节，避免被代码淹没。

### 1.2 Claude Code —— 双手（实施）

Claude Code 只做一件事：**拿一个 Task Shard，把它实现出来**。

- 每个 Task Shard 启动一个**独立的 Claude Code CLI 进程**。
- 该进程只载入这一个 Shard 的上下文（+ 必要的项目背景），**不复用其他 Shard 的上下文**。
- 实现完即退出，把实施记录交回 Codex 审查。

> 设计意图：这是 Loops 规避**上下文压缩**的核心手段——详见 §3。

### 1.3 人 —— 在环把关者

人在 Loop 中有三类介入：

1. **启动**：提交 Issue。
2. **门禁**：审核 Codex 生成的产品文档（通过才进入拆解实施）。
3. **随时介入**：在任意时刻手动新开一个 Claude Code CLI 或 Codex，做额外实施或修订（详见 [03-工作流 §6](03-工作流设计.md)）。

> Loops 不追求"无人化"，而是"**人只在关键节点出现**"。

## 2. 关键术语表

全文统一术语，后续文档不再重复定义。

| 术语 | 英文 | 定义 |
|------|------|------|
| **议题** | Issue | 用户提交的最顶层需求，一个 Loop 的输入。 |
| **产品文档** | Product Spec | Codex 由 Issue 生成的、经人审核的完整需求与方案文档。 |
| **任务碎片** | Task Shard | 产品文档被 Codex 拆解出的、自包含的最小实施单元；一个 Shard = 一个 Claude Code 实例。 |
| **循环** | Loop | 从"文档生成"到"收敛判定/回环/终态"的一个完整周期。 |
| **轮次** | Round | 一个 Loop 内的实施-审查迭代序号（Round 1、Round 2…）。 |
| **收敛** | Convergence | 所有 Shard 达到 DONE、必须测试 PASS、整体审查 PASS、文档全部标注的状态。 |
| **上下文隔离** | Context Isolation | 每个 Claude Code 实例只持有自己 Shard 的上下文，互不串扰。 |
| **上下文预算** | Context Budget | 单个 Shard 允许的最大上下文 token，超预算必须进一步拆解。 |
| **审查** | Review | Codex 对实施产出的正确性/完整性判断，结论为 PASS / NEEDS-WORK / FAIL。 |
| **测试矩阵** | Test Matrix | Codex 从 Spec/Shards 派生出的测试清单，覆盖单元、集成、端到端、回归、人工验收。 |
| **测试证据** | Test Evidence | Runner/Claude Code 执行测试后的结果、日志、覆盖率、失败原因与修复记录。 |
| **标注** | Annotation | 固化在文档/任务上的结构化状态元数据（实现状态、风险、覆盖度等）。 |
| **回环** | Re-loop | 整体复查发现问题后，Codex 重写文档、进入新一轮 Loop。 |
| **门禁** | Gate | 必须由人确认才能流转的检查点（默认：产品文档审核）。 |
| **编排引擎** | Orchestrator | Loops 的核心，驱动 Loop 各阶段、调度 agent、管理状态。 |
| **工作目录** | Workspace | `.loops/`，存放所有状态、产物、日志与标注。 |

## 3. 上下文压缩问题与 Loops 的解法（产品动机核心）

### 3.1 问题

Claude Code（以及所有长上下文 agent）在任务推进中，当累积上下文逼近模型上限时，会触发**上下文压缩**：早期对话、约束、已实现内容被摘要甚至丢弃。后果是：

- 遗忘早期约定（命名规范、架构约束、已修复的 bug）；
- 重复劳动或回退已完成的修改；
- 产出质量随轮次明显退化；
- 越往后越"失忆"，最终跑偏。

这是单 agent 处理大型任务的根本性瓶颈。

### 3.2 Loops 的解法：拆解 + 隔离

Loops **不试图喂给 agent 一个超大上下文**，而是反其道而行：

```
大文档(危险：会触发压缩)
        │  Codex 拆解（受上下文预算约束）
        ▼
Shard-1   Shard-2   Shard-3   ...   Shard-N      (每个都在预算内)
   │         │         │               │
   ▼         ▼         ▼               ▼
 CC进程1   CC进程2   CC进程3   ...   CC进程N        (独立、隔离、短命)
```

- **拆解**：Codex 按"上下文预算"把文档切成若干 Shard，保证每个 Shard 自包含且不超预算。
- **隔离**：每个 Shard 由一个全新的、短命的 Claude Code 进程实施，**进程结束上下文即销毁**——根本没有"长上下文"可供压缩。
- **纵览归大脑**：全局视图（哪些做了、哪些没做、整体一致性）始终留在 Codex 与 `.loops/` 状态里，不依赖任何单个实施进程的记忆。

> 一句话：**把"一个长记忆的大任务"变成"一堆无记忆的小任务 + 一个外部的全局状态"。** 压缩问题在架构层面被消解。

但这只解决了"单个 Claude Code 会话越来越长、越来越失忆"的问题，**不是完整质量保证**。如果 Shard 边界切错、接口契约不清、测试不足，短会话同样可能做错，只是错得更局部。

### 3.2.1 拆解能解决什么

| 能力 | 结论 |
|------|------|
| 降低单个 Claude Code 会话长度 | 能显著缓解 |
| 避免单会话上下文压缩导致遗忘 | 能显著缓解 |
| 让失败可局部重试 | 能显著改善 |
| 并行推进互不依赖的小任务 | 能提升吞吐 |

### 3.2.2 拆解不能单独解决什么

| 风险 | 原因 | 需要的补强 |
|------|------|------------|
| 跨 Shard 接口不一致 | 每个 Claude Code 只看局部 | 接口契约、依赖 DAG、全局复查 |
| 隐性依赖被漏掉 | 拆解时没识别共享状态/边界条件 | Codex 拆解自检 + 架构约束清单 |
| 多 Shard 改同一文件冲突 | 并发改动重叠 | 文件所有权、worktree、合并策略 |
| 局部实现通过但整体失败 | 单 Shard 验收不等于系统验收 | 集成测试、E2E、全量回归 |
| Shard 本身仍过大 | 任务估算偏小或实现反复迭代 | token 预算硬拦截、最大轮次、二次拆分 |
| Codex 审查漏判 | 只读摘要而非证据 | 测试证据、diff、失败日志、覆盖矩阵 |

因此 Loops 的更准确表述应是：**用拆解和隔离缓解上下文压缩，用契约和测试闭环保证最终质量。**

### 3.3 上下文预算的约束作用

上下文预算是一个**硬约束**，驱动拆解决策：

- 单个 Shard 的预估上下文 > 预算 → Codex 必须进一步拆分；
- 这保证了每个 Claude Code 进程都在"安全区"内运行，远离压缩阈值。

预算的取值、拆解维度与策略详见 [05-文档与任务模型 §3](05-文档与任务模型.md)。

## 4. 什么是"循环收敛"

Loops 的运行不是一条直线，而是一个**不断收敛的循环**：

```
Loop = 文档生成 → 审核 → 拆解 → 实施 → 审查+标注 → 收敛判定
                                                          │
                       ┌──────────────────────────────────┤
                       ▼                                  ▼
              [有剩余/未通过]                         [全部收敛]
              继续实施 / 回环重写                    终态标注 → 交付
```

**收敛（Convergence）的严格定义——三者同时成立：**

1. 所有 Task Shard 状态 = `DONE`（实现完成、必须测试通过且审查 PASS）；
2. 所有必须测试项 = `PASS`（单 Shard 测试、集成测试、回归测试按测试矩阵通过）；
3. Codex 整体复查 = `PASS`（跨 Shard 一致性、无遗留问题）；
4. 所有文档标注完整（实现状态、测试状态、审查结论、风险均已标注）。

只要未达收敛，Loop 就继续；过程中可随时由人新开 agent 加速或修正。**收敛是 Loop 唯一的正常出口。**

> 收敛保证了"杜绝半途而废"——没有"差不多行了"，只有"全部闭环"或"回环继续"。
