# API libs 业务与基础设施拆分方案

## 一、目标与原则

**目标**：把 libs 中「含业务功能」的代码与「纯技术/非业务」的代码在**目录与依赖关系**上拆开，便于：

- 业务逻辑集中、易演进与测试
- 基础设施可复用、可替换（换短信/邮件厂商等不影响业务层）
- 依赖方向清晰：**infra 不依赖 domain，domain 可依赖 infra**

**原则**：

- **基础设施（infra）**：与产品/领域无关，可在任意 Nest 项目中复用（装饰器、配置、HTTP 客户端、DB 连接、队列、缓存、工具函数、无业务规则的服务等）
- **业务（domain）**：与 Dofe 产品、领域规则、业务流程强相关（认证、用户/实体数据访问、消息、任务队列等；若将来拆分出纯业务应用服务，可放入 `domain/services`）

---

## 二、已落地的目录结构（infra 已提取至独立仓库）

当前 infra 已从 `libs/infra/` 提取至独立仓库 `infra.dofe.ai`，以 `@dofe/infra-*` npm 包形式提供。

## 三、依赖规则与边界（必须遵守）

### 3.1 依赖方向

- **infra 内部**：`@dofe/infra-*` 包之间可按需相互依赖（通过 peerDependencies），但**禁止**依赖 `libs/domain/*` 下任何内容。
- **domain 可依赖 infra**：`libs/domain/auth`、`libs/domain/services` 可以依赖 `@dofe/infra-*` 包（common、clients、prisma、redis、shared-db、shared-services 等）。
- **src（API 层）**：仅依赖 domain 与 infra，不被 libs 反向依赖。

整体依赖方向：**`src` → `domain` → `@dofe/infra-*`**。

### 3.2 边界检查清单

- [ ] **infra 不依赖 domain**：`@dofe/infra-*` 包内任何文件不得 `import` 来自 `libs/domain/**`。
- [ ] **domain 可依赖 infra**：domain 内可自由使用 `@app/shared-db`、`@app/prisma`、`@app/redis`、`@app/shared-services/*`、`@/config/*`、`@/utils/*` 等（这些别名通过 tsconfig 指向 node_modules 中的 @dofe/infra-* 包）。
- [ ] **新增代码归属**：新模块按「是否与 Dofe 领域/业务流程强相关」放入 domain 或贡献到 infra.dofe.ai 仓库，避免在 infra 中写业务规则。

## 四、路径别名（tsconfig）与导入约定

### 4.1 infra 相关别名（指向 node_modules/@dofe/infra-*）

| 别名                                                                  | 指向                                    | 用途                                                               |
| --------------------------------------------------------------------- | --------------------------------------- | ------------------------------------------------------------------ |
| `@/common/*`                                                          | `@dofe/infra-common/src/*`             | 装饰器、filter、guards、interceptor、middleware、pipes、ts-rest 等 |
| `@/config/*`                                                          | `@dofe/infra-common/src/config/*`      | 配置、校验、dto                                                    |
| `@/filter/*`                                                          | `@dofe/infra-common/src/filter/*`      | 异常过滤器                                                         |
| `@/decorators/*`                                                      | `@dofe/infra-common/src/decorators/*`  | 各类装饰器                                                         |
| `@/interceptor/*`                                                     | `@dofe/infra-common/src/interceptor/*` | 拦截器                                                             |
| `@/middleware/*`                                                      | `@dofe/infra-common/src/middleware/*`  | 中间件                                                             |
| `@/utils/*`                                                           | `@dofe/infra-utils/src/*`              | 工具函数                                                           |
| `@/prisma/*`、`@/prisma-read/*`、`@/prisma-write/*`                   | `@dofe/infra-prisma/src/prisma/*`      | Prisma 模块                                                        |
| `@app/shared-db`、`@app/shared-db/*`                                  | `@dofe/infra-shared-db/src`            | 事务基类、UnitOfWork                                               |
| `@app/prisma`、`@app/redis`、`@app/rabbitmq`、`@app/jwt`、`@app/i18n` | `@dofe/infra-*/src`                    | 对应 infra 包                                                      |
| `@app/clients/internal/*`、`@app/clients/plugin/*`                    | `@dofe/infra-clients/src/*`            | 客户端                                                             |
| `@app/shared-services/*`                                              | `@dofe/infra-shared-services/src/*`    | 共享服务（email、sms、file-storage 等）                            |

### 4.2 domain 相关别名

| 别名                       | 指向                           | 用途                     |
| -------------------------- | ------------------------------ | ------------------------ |
| `@app/auth`、`@app/auth/*` | `libs/domain/auth/src`         | 认证                     |
| `@app/db`、`@app/db/*`     | `generated/db/src`           | 自动生成的 DB Service 模块 |
| `@/db/*`                   | `generated/db/src/modules/*` | 按实体划分的 DB 模块路径   |

### 4.3 导入约定

- 在 **infra** 包内写代码时，使用相对路径引用同包内模块，使用 peerDependencies 引用其他 `@dofe/infra-*` 包，**不要**引用 `libs/domain/**`。
- 在 **domain** 内写代码时，可引用 infra 的任意别名；引用 domain 其他模块时使用 `@app/auth`、`@app/db` 等。

## 五、与四层架构的关系

- **不变**：仍保持 API 层 → Service 层 → DB 层 / Client 层；API 不直连 DB/外部 API；Service 经 DB Service 访库、经 Client 访外。
- **增强**：在「哪一层」之上，增加「放在 infra 还是 domain」的约束：
  - **infra**：技术能力、横切、无业务规则的服务与客户端（common、clients、prisma、redis、shared-db、shared-services 等），位于 `infra.dofe.ai` 仓库。
  - **domain**：与产品、领域、业务流程强相关的模块（auth、db 等）；若今后拆出纯业务应用服务，可置于 `domain/services`。

---

## 六、扩展与后续

- **infra.dofe.ai 贡献**：新增通用基础设施代码应贡献到 `infra.dofe.ai` 仓库，遵循其版本发布流程（Semantic Versioning）。
- **domain/services**：若将来把部分「纯业务应用服务」从 shared-services 拆出，可在 `libs/domain/services/` 下新增，并沿用「domain 可依赖 infra、infra 不依赖 domain」的边界。
- **规范引用**：本方案及 infra/domain 边界已写入 `CLAUDE.md`、`.cursorrules`，编码与 Code Review 时需遵守。
- **更多信息**：参考 `docs/0426/infra/` 目录下的设计方案、迁移检查清单与迁移状态文档。
