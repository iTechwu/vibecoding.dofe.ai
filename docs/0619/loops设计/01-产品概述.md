# 01 · 产品概述

## 1. 背景

AI 编码 agent（Claude Code、Codex、Cursor 等）已经能独立完成相当复杂的编码任务。但当任务规模变大、需要持续多轮推进时，三个结构性问题反复出现：

1. **上下文压缩（Context Compaction）**：单 agent 在长任务中会自动压缩历史上下文，导致它"忘记"早期决策、约束与已实现内容，产出质量随轮次退化。
2. **既写又审，缺乏校验**：同一个 agent 既写代码又审代码，等于自己给自己判卷，错误难以被发现。
3. **大任务半途而废**：大型需求无法在单次会话中完成，缺乏"持续推动直到收敛"的机制，人需要不断手动接力。

Loops 正是为这三个问题而生。

## 2. 问题陈述

> 如何让一个**大型编码需求（Issue）**，在**不依赖人持续手动接力**的前提下，被**高质量、可审查、可追溯**地实现到完全闭环，同时**规避单个 AI agent 的上下文压缩**？

Loops 的回答是：**用两个 agent 分工，用循环收敛，用文档标注锁定真相。**

## 3. 产品愿景

> **让 AI 自己把自己写完。**

Loops 致力于成为一款**自主、可信、闭环**的 AI 编码引擎：人只负责"提需求 + 关键审核"，剩下的拆解、实施、审查、标注、再循环，全部由 Codex 与 Claude Code 协作完成，直到所有任务收敛交付。

## 4. 目标用户

| 用户 | 场景 | 核心诉求 |
|------|------|---------|
| 独立开发者 | 用 AI 持续推进个人大型项目 | 不想手动接力，跑一晚上就有产出 |
| 小型工程团队 | 把需求拆给 AI 批量实施 | 可审查、可追溯、质量可控 |
| AI 重度用户 | 希望榨干多个 coding agent 的组合价值 | 双 agent 分工、规避单点退化 |
| 产品/技术负责人 | 希望看到 AI 推进的全过程 | 文档即真相、状态可观测 |
| 开发期使用者 | 在 Web 控制台随时提交需求、缺陷、优化点 | 低成本跑通 Issue → 文档 → 实施 → 审查闭环 |
| 飞书协作团队（后续） | 在聊天里随时抛出需求、缺陷、优化点 | 不切换工具也能进入 AI 编码闭环，且保留来源追溯 |

**非目标用户**：完全不写/不看代码、希望"一键全自动无审核"产出生产代码的用户（Loops 强调人在环审核）。

## 5. 核心价值主张

1. **规避上下文压缩** —— Codex 把大任务拆成小上下文，每个 Claude Code 实例专注一块，从源头避免压缩退化。
2. **交叉审查可信** —— Codex 审 Claude Code 的产出，双 agent 交叉校验，错误更早暴露。
3. **循环收敛到底** —— 剩余项持续推动，未通过就重写文档回环，直到全部闭环，杜绝半途而废。
4. **文档即真相** —— 每个产物都有状态标注，实施/审查/风险全程可追溯。
5. **入口即队列** —— Issue 不依赖 GitHub；开发期先以 Web 表单为唯一入口，后续扩展飞书消息、CLI、API，并完整记录提交用户、提交渠道、原始内容与后续确认链路。

## 6. 与现有方案的区别

| 方案 | 与 Loops 的区别 |
|------|----------------|
| **纯 Claude Code / 纯 Codex** | 单 agent 一条龙，长任务会压缩退化，既写又审无校验。Loops 双 agent 分工 + 循环。 |
| **Devin / Devika 等自主 agent** | 通常是单一大模型驱动、黑盒推进。Loops 显式拆解、显式循环、文档标注、人在环。 |
| **手动多开 agent** | 人肉编排、手动接力、状态易丢失。Loops 用编排引擎自动管理状态与循环。 |
| **Cursor / Windsurf IDE** | 面向"人主导、AI 辅助"的交互编辑。Loops 面向"AI 主导、人审核"的批量自主实施。 |

## 7. 设计原则

| 原则 | 含义 |
|------|------|
| **小上下文优先** | 拆解是第一公民；任何单次 agent 调用都应在可控 token 预算内，规避压缩。 |
| **角色分离** | 规划/审查（Codex）与实施（Claude Code）由不同 agent 承担，禁止同体。 |
| **循环收敛** | 不假设一次成功；以"收敛"为终止条件，未达则回环重写。 |
| **人在环** | 关键门禁（文档审核）由人把关；人可随时介入新开 agent。 |
| **文档即真相** | 状态、审查结论、风险都以标注形式固化在文档中，可审计。 |
| **来源可追溯** | 每个 Issue 必须记录 `source_channel`、`submitter`、`raw_payload`，后续 Spec、Shard、PR 均可追到最初入口。 |
| **统一身份** | 开发期与产品期均使用 Dofe SSO，`sso.dofe.ai` 负责登录态，`api.sso.dofe.ai` 负责用户信息与权限校验。 |
| **可中断 / 可恢复** | 任意环节可暂停、可恢复，状态持久化在 `.loops/`。 |
| **失败可见** | 任何 agent 失败、超时、未通过都明确记录，不静默吞掉。 |

## 8. 非目标（明确不做什么）

- **不替代人做最终生产决策** —— 人仍是最终把关者，Loops 不主张"无人可信交付"。
- **不做一个通用 chat agent** —— Loops 是"任务编排引擎"，不是对话助手。
- **不把 GitHub Issue 作为唯一入口** —— GitHub/代码托管平台只是可选同步目标；开发期一等入口是 Web 控制台，飞书后续接入。
- **不绑定单一模型厂商** —— 角色可替换（Codex 可换其他规划 agent，Claude Code 可换其他实施 agent），接口抽象。
- **不做 CI/CD 替代品** —— 不接管部署流水线，专注"需求 → 实现 → 审查 → 标注"的编码闭环。

## 9. 产品化形态

Loops 从本地工具升级为面向团队的产品后，默认采用 **NestJS + Next.js** 的前后端分离形态。开发期先收敛为 **SSO 登录 + Web 页面提交 Issue + Web 审核台**，降低飞书 Webhook 与企业应用配置的前置复杂度。

| 层 | 形态 | 责任 |
|----|------|------|
| Next.js Web 控制台 | SSO 登录、Issue 提交、审核台、Issue 队列、Loop 看板、标注看板、人工介入入口 | 让产品/技术负责人提交、审核、追踪、接管 |
| NestJS API / Orchestrator | SSO 校验、Issue Intake、Loop 状态机、任务调度、agent 适配 | 承载核心编排逻辑 |
| Worker / Runner | Codex CLI、Claude Code CLI、git worktree、测试命令执行 | 隔离执行高风险和长耗时任务 |
| 文件工作区 + 数据库 | `.loops/` 文档真相源 + PostgreSQL/SQLite 索引 | 兼顾可读文档与产品查询 |
